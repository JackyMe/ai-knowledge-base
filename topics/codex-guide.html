认识 00

Codex 全景

Codex 是 OpenAI 的代理式编码工具:读代码、跑命令、改文件、自主推进任务。与 Claude Code 同属"终端编码代理"物种,心智模型高度相通(上下文管理、计划先行、验证闭环同样适用),但形态矩阵与安全模型有自己的鲜明取向。

五种形态,一个账号

形态入口适用
CLI(TUI)终端运行 codex日常主力:交互式终端 UI
非交互模式codex exec(别名 codex e)脚本与 CI:结果流式输出到 stdout 或 JSONL
IDE 扩展VS Code / Cursor 等安装 Codex 扩展编辑器内对话、内联改代码
Codex Cloudcodex cloud / 网页端云端沙箱跑任务,支持 best-of-N 多次尝试,回本地 codex apply 应用 diff
桌面 App / GitHubcodex app;GitHub 中 @ 提及独立窗口工作台;PR 自动代码评审

模型与推理力度

会话内用 /model 切换模型并选择推理力度(reasoning effort),启动时用 --model 指定(官方文档示例如 gpt-5.4;具体型号随版本演进,以 codex debug models 输出为准)。/fast 可切换 Fast 服务档;--oss 甚至能接本地 Ollama 开源模型。

一句话定位(社区共识)Codex 的招牌是默认更严格的沙箱模型(命令在 OS 级沙箱里跑)与云端任务矩阵;Claude Code 的招牌是更丰富的扩展面(Skills/Hooks/子代理生态)。两者高度互补,见协同一节。

认识 01

安装与登录

安装

# npm 方式
npm install -g @openai/codex

# 或 Homebrew(macOS)
brew install codex

# 启动 / 自更新
codex
codex update

登录

codex login 支持三种方式:ChatGPT 账号 OAuth(推荐,Plus/Pro/Team 订阅直接用)、设备码认证、或从 stdin 传入 API Key。登出用 codex logout;会话内 /logout。

进入新仓库的标准四步

  1. /init —— 生成 AGENTS.md 脚手架(构建命令、测试方式、项目约定)
  2. /model —— 选定模型与推理力度;/status 确认会话配置与 token 用量
  3. 确认沙箱与审批档位(见 03 沙箱与审批)——这是 Codex 使用体验的核心开关
  4. /mcp —— 查看已接入的外部工具;需要就 codex mcp add

配置体系:config.toml 三层

层级位置说明
全局~/.codex/config.toml个人默认:模型、沙箱、审批策略、MCP、profiles
项目.codex/config.toml(仓库根)覆盖全局,冲突时项目优先
单次运行-c key=value / --profile 名称临时覆盖任意配置;profile 可打包一组预设
# ~/.codex/config.toml 常用起步项
#:schema https://developers.openai.com/codex/config-schema.json
model = "gpt-5.4"
sandbox_mode = "workspace-write"     # 沙箱默认档
approval_policy = "on-request"       # 审批默认档
编辑体验文件顶部加上面那行 schema 注释,配合 VS Code 的 Even Better TOML 扩展即可获得自动补全与校验。诊断配置层叠用会话内 /debug-config。

速查 03

沙箱与审批:Codex 的安全内核

Codex 的安全模型是两个正交的旋钮:沙箱决定"命令能碰什么",审批决定"什么时候要问你"。理解这 3×3 组合,就掌握了 Codex 的使用手感。

沙箱三档(--sandbox / sandbox_mode)

档位能力适用
read-only只能读,不能写不能联网代码调研、评审、问答
workspace-write 推荐默认可写工作区(及 --add-dir 追加目录),网络默认受限日常开发
danger-full-access无沙箱限制仅限容器等隔离环境

审批三档(--ask-for-approval / approval_policy)

档位行为适用
untrusted非白名单命令一律先问陌生仓库、高危操作期
on-request 推荐默认模型自行在沙箱内干活,越权需求才来问你交互式日常(官方推荐,on-failure 已弃用)
never从不询问,越权直接失败CI 与无人值守脚本
--yolo 的正确认知--dangerously-bypass-approvals-and-sandbox(别名 --yolo)= 审批与沙箱全关。与 Claude Code 的 --dangerously-skip-permissions 同理:只应出现在无敏感数据、无网络副作用的外部加固环境(如一次性容器)。

进阶:execpolicy 规则

codex execpolicy 可以评估自定义规则文件,预演某条命令会被放行、询问还是拦截——把团队的命令安全策略做成可测试的代码。另有 codex sandbox 让你把任意命令手动丢进 Codex 同款 OS 沙箱里跑,适合验证沙箱行为。

来源:官方 CLI Reference · SmartScope Codex Guide

配置与扩展 04

AGENTS.md 与记忆

AGENTS.md:开放标准的项目说明书

作用等价于 Claude Code 的 CLAUDE.md:每次会话自动加载的项目级指令。差异在于 AGENTS.md 是跨工具开放标准(agents.md),Codex、Cursor 等多家代理都认它——写一份,多个工具受益。

  1. /init 生成脚手架:构建命令、测试方式、项目约定
  2. 手工打磨:留下"猜不出来的事实",删掉读代码就能知道的内容(精简法则与 CLAUDE.md 完全一致)
  3. 子目录可放自己的 AGENTS.md,作用于该子树——monorepo 友好

配置与账号体系全解

把 ~/.codex 里的每个文件、每种登录形式、换电脑/无头机的每条路径讲透——按"是什么 → 在哪里 → 怎么改 → 出问题怎么查"组织。

一、CODEX_HOME 目录地图

Codex 的所有本地状态都在 CODEX_HOME(默认 ~/.codex,设该环境变量可整体搬家):

文件 / 目录内容要不要备份
config.toml核心配置:模型、沙箱、审批、MCP、特性旗标、模型提供方✅ dotfiles 必备
<名称>.config.toml配置档(profile)文件,与 config.toml 同目录,--profile 名称 启用✅ 一起带走
AGENTS.md全局个人指令(项目层的在仓库里)✅
auth.json登录凭证缓存(明文 token!)❌ 当密码对待,不进 git 不进聊天
日志目录(含 codex-login.log)运行与登录诊断日志无需备份;排查登录问题时看它
会话数据codex resume / fork 依赖的本地会话记录本地资产,不随机器走

二、配置层级与 Profile

同一个键冲突时,从高到低:

  1. 企业管控 —— managed configuration 与 requirements.toml 约束,个人不可越过
  2. 单次覆盖 —— -c key=value(值按 JSON 解析)
  3. Profile —— --profile 名称 加载对应 名称.config.toml
  4. 项目层 —— 仓库根的 .codex/config.toml(注意:出于安全,model_providers、notify、otel 等键在项目层被忽略,必须写在用户层)
  5. 用户层 —— ~/.codex/config.toml
  6. 内置默认值
诊断三板斧/status 看当前会话生效的模型/审批/可写根;/debug-config 打印配置层叠与 requirements 诊断("为什么这个键不生效"一查便知);config.toml 顶部加 #:schema https://developers.openai.com/codex/config-schema.json 获得编辑器校验。

三、登录方式全解与切换

方式怎么用计费 / 适用
ChatGPT 登录(默认)codex login → 浏览器 OAuth(本机回调 localhost:1455)订阅额度;Fast mode 等依赖 ChatGPT 额度的功能仅此方式可用;走 ChatGPT 工作区的权限与数据政策
API Key平台后台取 key,经 stdin 传给 codex login按量 API 计费;官方推荐 CI/CD 用这种;走 API 组织的数据设置
设备码(beta)codex login --device-auth,或登录界面选 Sign in with Device Code;需先在 ChatGPT 安全设置(个人)或工作区权限(管理员)中启用无浏览器/回调被挡的环境首选
社交/SSO 登录Google / Microsoft / Apple / 企业 SSO 直接走 ChatGPT 登录页MFA 建议在提供方处开启;邮箱+密码账号必须开 MFA 才能用 Codex cloud

凭证存到哪、怎么切换

  • 存储位置可选:config.toml 里 cli_auth_credentials_store = "file" | "keyring" | "auto"——file 存明文 auth.json;keyring 存系统凭据库(macOS 钥匙串等);auto 优先系统凭据库。
  • CLI 与 IDE 扩展共享同一份缓存:任一端 logout,另一端也要重登;ChatGPT 会话的 token 会自动续期。
  • 切换账号 = codex logout → codex login 选新账号;会话内 /logout 同效。
  • 企业强制:管控配置可设 forced_login_method = "chatgpt" | "api" 与 forced_chatgpt_workspace_id;凭证不匹配会被直接登出退出——"公司机器换不了登录方式"就是它。

四、换电脑 / 无头机 / 远程环境

普通换电脑

# 新机三步
npm install -g @openai/codex     # 或 brew install codex
codex login                      # 浏览器重新登录(凭证不建议拷贝)
# 把 dotfiles 里的 config.toml / profiles / AGENTS.md 放回 ~/.codex

无头机 / 服务器 / 容器(官方三方案,按优先级)

  1. 设备码认证(首选):codex login --device-auth → 本地浏览器开链接输一次性码。
  2. 拷贝 auth.json(官方兜底):在有浏览器的机器登录后,把缓存复制过去—— 
    # SSH
ssh user@remote 'mkdir -p ~/.codex && cat > ~/.codex/auth.json' < ~/.codex/auth.json
# Docker 容器
docker cp ~/.codex/auth.json 容器名:/root/.codex/auth.json
    注意:若本机凭证存在系统钥匙串(keyring)而非文件,此法不适用——先把 cli_auth_credentials_store 设为 file。
  3. SSH 端口转发:ssh -L 1455:localhost:1455 user@remote 后在远端跑 codex login,用本地浏览器完成标准流程。
安全红线auth.json 就是你的账号:含访问 token,不 commit、不贴工单、不发聊天。CI 长期方案优先 API Key 存 Secret,而不是搬运 auth.json。

企业网络(TLS 代理 / 私有 CA)

export CODEX_CA_CERTIFICATE=/path/to/corporate-root-ca.pem
codex login   # 登录、HTTPS、WebSocket 全部走该 CA;未设时回退 SSL_CERT_FILE

五、第三方模型与网关

在用户层 config.toml 定义 model_providers.<id> 后,三种认证任选其一:

方式写法场景
OpenAI 认证过网关requires_openai_auth = true(此时 env_key 被忽略)经 LLM 代理/网关访问 OpenAI 模型,仍用 ChatGPT/API Key 登录

六、高频问题急救表

症状原因 / 解法
浏览器登录卡住 / 回调失败本机 1455 端口被占或被防火墙拦:改用 --device-auth,或 SSH 转发 1455
登录问题要证据看日志目录里的 codex-login.log(login 专用日志)

进阶 07

云端与多形态

Codex Cloud:云沙箱跑任务

# 终端直接派发云任务(--env 指定环境,--attempts 跑 best-of-N)
codex cloud exec --env ENV_ID --attempts 3 "重构 utils/ 并补齐测试"

# 浏览任务列表(--json 机器可读)
codex cloud list --limit 10

# 满意的结果一键落地到本地工作树
codex apply TASK_ID

best-of-N 是云端的杀手锏:同一任务并行跑 1–4 次尝试,挑最好的 diff 应用——对"一次成型率不稳"的大任务尤其划算。

IDE 扩展与桌面 App

  • VS Code / Cursor 安装 Codex 扩展:编辑器里对话、内联 diff、把当前文件上下文喂给代理;
  • codex app 启动桌面工作台(macOS 可直接带工作区路径打开);
  • --remote ws://… 让终端 TUI 连接远程 app-server——本地键盘,远端算力。

GitHub 代码评审

接入 GitHub 后,在 PR 中 @ 提及 Codex 即可触发自动代码评审;被自动评审拦下的重试可在 CLI 用 /approve 放行一次。

进阶 08

实战场景手册

🧭场景① 只读调研模式EXPLORE

codex --sandbox read-only "梳理这个仓库的模块依赖关系"——物理只读,放心让它翻遍整个代码库。配合 /mention 精确指定要看的文件,/plan 让它先出方案再动手。

🏗️场景② 日常开发标准流FEATURE

推荐组合 workspace-write + on-request:模型在沙箱里自主干活,越权才打断你。链路:/plan 出计划 → 确认后实现 → /diff 审查改动 → /review 让它自评一轮 → 提交。/goal 把验收条件挂到会话上。

🤖场景③ 自动化与 CIAUTOMATION
# 非交互执行:JSONL 输出便于管道处理
codex exec --sandbox workspace-write -a never \
  "修掉所有 eslint 报错并跑通测试" --json

# 批量:恢复既往会话续跑
codex exec resume SESSION_ID "继续处理剩余文件"

要点:CI 里固定 -a never(越权即失败,不挂起等人),用 execpolicy 规则文件约束可执行命令。

☁️场景④ 大任务上云,好结果落地CLOUD

本地继续写代码,大重构丢给云:codex cloud exec --attempts 3 并行三次尝试 → 网页/终端查看各次 diff → codex apply 把最优解落到本地 → /diff + 本地测试验收。

🔀场景⑤ 双代理协同(Codex × Claude Code)DUAL-AGENT

三种玩法:① /import 一键把 Claude Code 的配置、项目文件、近期会话搬进 Codex,无痛双开;② Writer/Reviewer 跨代理——一个写实现,另一个 /review,不同模型的视角差异比同模型双会话更能暴露盲区;③ 互为 MCP 工具(见 05 节),编排层随意转包。技能包(Agent Skills 标准)两边通用,工作流沉淀一次即可。
