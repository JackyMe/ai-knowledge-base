心法 00

一切围绕上下文

Claude Code 不是聊天机器人,而是代理式编码环境:它读你的文件、跑你的命令、改你的代码、自主推进问题,你负责观察、纠偏或者干脆走开。官方 Best Practices 开篇就点破了几乎所有实践背后的那个约束——

核心约束上下文窗口装着整场对话:每条消息、每个读过的文件、每次命令输出。它填满得很快,而模型性能随之下降——上下文一满,Claude 会开始"忘记"早先的指令、犯更多错。上下文是你最需要管理的资源,没有之一。

由这一条可以推出本指南的大部分结论:

  • 为什么要用子代理做调研 → 探索消耗的是它们的上下文,不是你的;
  • 为什么 CLAUDE.md 要短 → 每个会话都加载,太长反而让规则被噪音淹没;
  • 为什么 Skills 优于往 CLAUDE.md 堆知识 → 技能按需加载,不用不花钱;
  • 为什么要 /clear、/compact、/btw → 都是给上下文腾地方的工具;
  • 为什么"纠正两次就重开" → 失败尝试也留在上下文里,会把模型往同一个坑里带。

五种形态,同一个引擎

Claude Code 不只活在终端里。关键在于:所有形态共用同一引擎,CLAUDE.md、settings、MCP 配一次处处生效。

心法 01

五条黄金法则

官方 Best Practices 数千字,可以浓缩为五条。每一条都值得贴在显示器边上。

法则一:给 Claude 一个能自己跑的验证

Claude 在"看起来完成"时就会停手。没有可运行的检查,你就是那个验证环节——每个错误都要等你发现。给它一个能产出 pass/fail 的信号(测试套件、构建退出码、linter、对照截图),循环就自动闭合:干活 → 跑检查 → 读结果 → 迭代到通过。

❌ 弱"实现一个校验邮箱的函数" / "构建挂了,修一下"
✅ 强"写 validateEmail,测试用例:user@example.com→true、invalid→false、user@.com→false,实现后跑测试" / "构建报错如下[贴错误],修复并验证构建通过,解决根因,不要压掉报错"

验证的四个强度档位,按无人值守程度递进:同一条提示里要求"跑检查并迭代" → 用 /goal 设定条件让评估器每回合复查 → 用 Stop hook 把检查做成硬门禁(不通过就不许结束回合) → 让验证子代理用全新上下文给出第二意见。另外:要求 Claude 出示证据(测试输出、命令返回、截图),而不是口头宣布成功。

法则二:先探索、再计划、后写码

直接开写,很容易解决了一个错误的问题。推荐四段式:Explore(计划模式下读文件、问问题,不改代码)→ Plan(产出实施计划,Ctrl+G 可在编辑器里直接改计划)→ Implement(退出计划模式实现,对照计划自验)→ Commit(写清晰的提交信息、开 PR)。

别把计划当教条范围明确的小改动(改错字、加日志、重命名)直接让它干。官方的判断标准:如果一句话能说清 diff 长什么样,就跳过计划。计划最值钱的时机:方案不确定、涉及多文件、你对这块代码不熟。

法则三:提示要具体,上下文要够料

Claude 能推断意图,但不能读心。指名文件、说明约束、指向既有范式:

策略弱提示强提示
圈定范围"给 foo.py 加测试""为 foo.py 写测试,覆盖用户已登出的边界情况,不要用 mock"
指向信源"这 API 怎么这么怪?""翻 ExecutionFactory 的 git 历史,总结这个 API 是怎么演化成现在这样的"
引用范式"加个日历组件""参考首页现有 widget 的实现模式(HotDogWidget.php 是好例子),照此实现日历组件……不引入代码库之外的新依赖"
描述症状"修登录 bug""用户反馈会话超时后登录失败。查 src/auth/ 的认证流,重点看 token 刷新。先写一个能复现的失败测试,再修"

喂料的五种姿势:@文件 引用(Claude 会先读再答)、直接粘贴图片(Ctrl+V)、给 URL(配合 /permissions 放行常用域名)、管道灌入(cat error.log | claude)、或者让它自己去取(跑命令 / MCP / 读文件)。

法则四:小步纠偏,勤清上下文

  • Esc 随时打断,上下文保留,可立即改方向;
  • Esc Esc / /rewind 打开回滚菜单:恢复对话、恢复代码、或两者都恢复;
  • "Undo that" 让它撤销自己的修改;
  • /clear 在不相关任务之间必用;同一问题纠正两次仍错,别恋战——/clear 后用吸收了教训的更好提示重来,几乎总是赢过一个塞满失败尝试的长会话。

法则五:让它先采访你(大功能必备)

大功能别急着下指令,先让 Claude 用 AskUserQuestion 工具反过来采访你:

我想做[简要描述]。用 AskUserQuestion 工具详细采访我:
技术实现、UI/UX、边界情况、顾虑与取舍。别问显而易见的问题,
挖我可能没想到的硬骨头。问完把完整规格写入 SPEC.md。

规格定稿后开新会话执行:干净的上下文专注实现,SPEC.md 是随时可查的锚点。好规格自带三要素:点名涉及的文件与接口、写明什么不做、以一个端到端验证步骤收尾。

速查 02

安装与首次配置

🚑 装不上 / 登录报错 / 换电脑?安装登录环节的一切疑难(凭证打架、共用电脑、SSH/CI、万能重置)在独立专题:环境与登录急救手册,按症状 30 秒直达修复。

安装与登录

# 安装(npm 方式)
npm install -g @anthropic-ai/claude-code

# 启动并登录(浏览器 OAuth)
claude
claude auth status        # 查看登录状态(JSON;--text 人类可读)

# 更新与体检
claude update             # 升级到最新版
claude doctor             # 安装与配置诊断(只读,不进会话)

进入新仓库的标准五步

  1. /init —— 分析代码库(构建系统、测试框架、代码模式)生成 CLAUDE.md 初稿,之后用 /memory 持续打磨
  2. /permissions —— 配好 allow/ask/deny 规则,把 npm run lint、git diff 这类安全命令放进白名单(或直接跑 /fewer-permission-prompts 让它根据你的历史自动生成白名单)
  3. /mcp —— 接上项目需要的外部工具(数据库、Figma、Notion、Sentry……)
  4. 安装 gh CLI —— Claude 天生会用它建 issue、开 PR、读评论;没有它就只能裸调 API 还常撞限流
  5. /statusline —— 配置状态栏持续显示上下文用量(法则零:时刻知道自己还剩多少上下文)
CLAUDE.md 的取舍标准(官方)逐行自问:"删掉这行,Claude 会犯错吗?"不会就删。该写:猜不出来的构建命令、异于默认的代码风格、测试跑法、分支/PR 约定、项目特有的架构决策、环境坑点。不该写:读代码就能知道的事、语言常识、详细 API 文档(给链接)、频繁变化的信息、"把代码写好"之类的废话。文件过长的直接后果:重要规则被淹没、指令被无视。可用 @path/to/file 语法导入其他文件;放置层级:~/.claude/CLAUDE.md(全局)→ 项目根(团队共享,进 git)→ CLAUDE.local.md(个人,进 .gitignore)→ 子目录(按需加载)。

扩展体系 05

六大扩展怎么选

Claude Code 的扩展面有六层。选错层是新手最大的内耗来源——把该进 Hook 的规则写进 CLAUDE.md 祈祷它遵守,或把偶尔才用的领域知识塞满每个会话。一张决策表说清:

你的需求用哪层核心理由
每个会话都必须知道的简短事实(构建命令、代码约定)CLAUDE.md每会话自动加载——所以必须短
特定场景才用的领域知识 / 重复的多步流程Skills按需加载,不用不占上下文;可 /名称 手动调用
必须 100% 发生的规则(强制 lint、禁写敏感文件)Hooks确定性执行。提示词是"请求",Hook 是"保证"
大量读文件的调研 / 需要独立视角的评审子代理独立上下文窗口,探索不污染主对话
连接外部系统(数据库、Figma、Slack、监控)MCP标准协议接工具,一次配置处处可用
打包分发以上一切给团队插件skills+hooks+子代理+MCP 一键安装;/plugin 逛市场

扩展体系 06

Skills / Hooks / MCP / 子代理:深入四件套

Skills:按需加载的专家技能包

一个技能 = 一个目录 + 一个 SKILL.md(可附模板、示例、脚本)。Claude 匹配到场景自动加载,或你 /技能名 手动调用。判断该建技能的信号:你在反复粘贴同一段说明/清单/流程,或 CLAUDE.md 里某段已经从"事实"长成了"流程"。

.claude/skills/my-skill/
├── SKILL.md        # 必需:frontmatter + 指令正文
├── examples/       # 可选:期望输出示例
└── scripts/        # 可选:可执行脚本
存放位置生效范围
~/.claude/skills/个人,所有项目
.claude/skills/(项目)本项目(进 git 即团队共享);monorepo 子目录也会按需发现
插件内 skills/插件启用处,命名空间 插件名:技能名

三个高级机关:

  • 动态上下文注入:正文里写 !`git diff HEAD`,加载技能前先执行命令、把输出内联进指令——技能天生"知道现场";
  • frontmatter 控制权:disable-model-invocation: true 让危险流程(部署等)只能手动触发;description 写成"路由规则"的口吻决定自动触发准确率;
  • 技能目录热更新:改盘上文件当场生效;同名优先级 企业 > 个人 > 项目 > 捆绑技能(所以项目里放个 code-review 技能可覆盖官方版)。
写好技能的社区套路description 像路由规则一样写清"何时用我";确定性的工作交给附带脚本而不是靠模型发挥;SKILL.md 保持精瘦、细节推到按需加载的附属文件;一个技能只干一件事;给具体的正反示例而不是抽象规则堆。

Hooks:确定性的护栏

CLAUDE.md 和技能里的指令是建议;必须发生的事,属于 Hook。Hook 在生命周期事件上运行脚本,以退出码 2 即可阻断动作。常用事件:

事件时机典型用法
PreToolUse工具调用前(可阻断)禁写 .env/密钥文件、拦下危险命令
PostToolUse工具调用成功后每次编辑后自动跑 prettier/eslint
StopClaude 准备结束回合跑测试,不过就不许收工(连续阻断 8 次后放行)
UserPromptSubmit你提交提示后、Claude 处理前注入环境信息、敏感词检查
SessionStart / SessionEnd会话起止初始化环境 / 清理与上报
PostToolUseFailure / PreCompact / SubagentStop / FileChanged…更细粒度事件失败告警、压缩前存档、监听文件变化等(30+ 事件见官方 Hooks 参考)
// .claude/settings.json —— 每次编辑后自动格式化
{
  "hooks": {
    "PostToolUse": [
      { "matcher": "Write|Edit",
        "hooks": [ { "type": "command", "command": "npx prettier --write \"$FILE\"" } ] }
    ]
  }
}
偷懒的正确方式Hook 可以让 Claude 替你写:"写一个 hook,每次文件编辑后跑 eslint"、"写一个 hook,阻止对 migrations 目录的写入"。/hooks 查看已配置项。
比 Hook 更安全的做法敏感文件防读,权限规则优于 Hook:deny 规则(如 Read(./.env)、Read(./secrets/**))让文件对 Claude 直接"不可见"。权限评估顺序:deny → ask → allow。

子代理(Subagents):独立上下文的分身

定义在 .claude/agents/*.md,拥有自己的上下文窗口、工具集与系统提示。两大用途:

  • 调研隔离:"用子代理调查认证系统如何处理 token 刷新"——它读几十个文件,你的主上下文只收到一份结论摘要;
  • 无偏评审:全新上下文看 diff,不被"刚写完这段代码的思路"带偏。官方建议的对抗式评审提示:点名要查的工作、对照的计划、什么算问题——"报告缺口,不报风格偏好"。
# .claude/agents/security-reviewer.md
---
name: security-reviewer
description: 代码安全评审。代码变更后主动使用。
tools: Read, Grep, Glob, Bash
model: opus
---
你是资深安全工程师。审查:注入类漏洞(SQL/XSS/命令注入)、
认证与越权缺陷、代码中的密钥、不安全的数据处理。
给出具体行号引用与修复建议。
/btw 与子代理互为镜像/btw 看得见全部对话但没有工具(问"已知的事");子代理有全部工具但从空上下文出发(去"查新的事")。选对方向,省一半上下文。
评审代理的通病被要求找问题的评审者总能"找到"问题——哪怕工作没毛病。追着每条发现改,结局是过度设计。给评审者立规矩:只报影响正确性或既定需求的缺口,其余当可选项。

MCP:连接外部世界

MCP(Model Context Protocol)让 Claude Code 直连数据库、Figma、Slack、Sentry 等外部系统。配置不在 settings.json 里——这是最常见的迷路点。

添加与管理

# 添加本地 stdio 服务器
claude mcp add my-tool -- npx -y some-mcp-server

# 添加远程 HTTP 服务器
claude mcp add --transport http api-tool https://example.com/mcp

# 需要 OAuth 的服务器单独走认证
claude mcp login api-tool

# 会话内:/mcp 查看状态、认证、启停某个服务器

三种作用域(--scope)

作用域存放位置适用
local(默认)~/.claude.json 的当前项目条目只有你、只在本项目用
project仓库根 .mcp.json,进 git团队共享——项目必备工具写这里
user~/.claude.json你的所有项目通用(如浏览器、笔记工具)

三条实用纪律

  • 工具也占上下文:每个 MCP 服务器的工具定义都会进窗口,接得越多越挤——/context 查占用,不用的服务器在 /mcp 里禁用;
  • 把 MCP 内容当不可信输入:外部工具返回的文本可能含注入指令,授权时只给必需的服务器;
  • 用命令别手写:claude mcp add 会写对格式;手改 JSON 出错是"MCP 消失"的常见原因。

八大场景实战手册

🧭场景① 接手陌生代码库ONBOARDING

像问资深同事一样直接问,不需要任何特殊提示:"日志是怎么打的?"、"新建一个 API endpoint 的流程?"、"foo.rs 第 134 行的 async move 是干嘛的?"、"这里为什么调 foo() 不调 bar()?"。想追历史就指向 git:"翻这个模块的提交历史,总结它为什么长这样"。

组合拳:大范围摸底用子代理("用子代理梳理整个支付模块的调用链"),主上下文只留结论;顺手 /init 把摸清的约定沉淀进 CLAUDE.md。

🏗️场景② 开发新功能FEATURE

标准链路:采访出 SPEC(法则五)→ 新会话 → /plan 探索与规划 → Ctrl+G 编辑计划 → 退出计划模式实现 → 对照验证 → 提交开 PR。

关键细节:实现指令自带验证("实现计划中的 OAuth 流程,为 callback handler 写测试,跑测试并修复失败");用 /goal 测试全绿 把验收条件挂到会话级;完成后 /code-review + "用子代理对照 SPEC.md 检查缺口"双保险。

🐛场景③ 调试与修 BugDEBUG

黄金模板:症状 + 疑似位置 + "修好"的定义——"用户反馈会话超时后登录失败。查 src/auth/ 的 token 刷新。先写一个复现问题的失败测试,再修,并验证测试通过"。

提效招:报错直接管道灌入 cat error.log | claude -p "分析根因";跑测试用 ! npm test(shell 直通,输出自动入上下文并得到解读);要求"解决根因,不要压掉报错";修完让它出示测试输出作为证据。

🔍场景④ 代码评审REVIEW

四把刀按需选:/code-review(本地 diff 找 bug,--fix 直接修)、/simplify(只做清理:复用/化简/去冗余)、/security-review(安全专项)、/review 123(快速过 GitHub PR)。深度需求上 /code-review ultra 走云端多代理。

双会话模式:A 会话写代码,B 会话(全新上下文)评审——"审查 @src/middleware/rateLimiter.ts,找边界情况、竞态,以及与现有中间件模式的一致性",再把 B 的产出喂回 A 修复。写测试同理:一个写测试,另一个写实现。

🚚场景⑤ 大规模重构 / 迁移MIGRATION

交互式:一句 /batch 把 src/ 从 Solid 迁到 React——自动拆解 5–30 个独立单元、每单元后台子代理 + 独立 worktree、各自实现-测试-开 PR。

脚本式(fan-out):先让 Claude 列出全部待迁移文件 → 挑 2–3 个试跑调好提示 → 全量循环:

for file in $(cat files.txt); do
  claude -p "把 $file 从 React 迁到 Vue,返回 OK 或 FAIL" \
    --allowedTools "Edit,Bash(git commit *)"
done

要点:--allowedTools 收窄权限、输出约定成机器可判(OK/FAIL)、先小样再全量。

🤖场景⑥ 自动化与 CIAUTOMATION

claude -p 是把 Claude 塞进任何管道的接口:

claude -p "列出所有 API endpoint" --output-format json
claude -p "分析这份日志" --output-format stream-json --verbose
claude --permission-mode auto -p "修掉所有 lint 报错"

🧵场景⑦ 并行与后台作业PARALLEL

单线程忙不过来时的四个层级:worktree 并行(claude -w 每个任务独立检出,互不踩踏)→ 后台化(/bg 转后台、/fork 派分身办侧线任务、Ctrl+B 后台化长命令)→ 后台舰队(claude agents 总控台:attach / logs / respawn)→ Agent Teams(带队长、共享任务与消息的多会话自动协作)。

诊断:先看清再动手

四种腾地方的手段(由轻到重)

手段动作适用时机

实战 09

权限与安全

六种权限模式

模式行为适用
default敏感操作逐一询问默认;不熟悉的任务
acceptEdits文件编辑自动放行,命令仍询问信任度中等的重构
plan只读:能看能想,不能改探索、规划、评审
auto分类器模型代替你审批,只拦可疑动作(越权扩散、未知基础设施、被恶意内容驱动的行为)方向已定、不想逐条点头的任务
dontAsk不询问,不放行的直接跳过特定自动化管道
bypassPermissions全部放行,零门禁⚠️ 仅限无网络隔离环境

Shift+Tab 在模式间循环;--permission-mode 指定启动模式。

规则三层:deny → ask → allow

// .claude/settings.json —— 一份务实的起步配置
{
  "permissions": {
    "allow": [
      "Read", "Glob", "Grep",
      "Bash(npm run lint:*)", "Bash(npm run test:*)",
      "Bash(git status)", "Bash(git diff:*)"
    ],
    "ask":  [ "Bash(git push:*)" ],
    "deny": [
      "Read(./.env)", "Read(./.env.*)", "Read(./secrets/**)"
    ]
  }
}
  • deny 最先评估:敏感文件放 deny,对 Claude 直接不可见——比用 Hook 拦更彻底;
  • 审批疲劳是安全隐患:点第十次"允许"时你已经不在审查了。解法是白名单(/fewer-permission-prompts 可自动生成)+ /sandbox OS 级隔离,而不是硬扛;
  • --dangerously-skip-permissions 只该出现在无网络、无敏感数据的封闭环境(如容器里跑 lint 修复);企业可用 disableBypassPermissionsMode 全组织禁用。

三、优先级与"为什么没生效"

同一个键冲突时:企业管控 > 命令行参数 > .claude/settings.local.json > .claude/settings.json > ~/.claude/settings.json。三件事最常造成"改了没反应":

  • 改错了层:项目层悄悄覆盖了你的用户层 → /config 看当前生效值,或用 --setting-sources user 只加载指定层排除干扰;
  • 该键要重启:大多数设置热加载,但 model、outputStyle 等在会话开始时读取——改完开新会话(临时切模型用 /model);
  • 配置本身坏了:claude doctor(终端体检)/ /doctor(会话内)会报 schema 错误;settings.json 顶部加 "$schema": "https://json.schemastore.org/claude-code-settings.json" 编辑器直接标红。怀疑自定义配置搞坏一切时,claude --safe-mode 全裸启动逐项排除。

云厂商直连CLAUDE_CODE_USE_BEDROCK=1 + /setup-bedrock 向导;Vertex 同理 /setup-vertex走 AWS/GCP 账单与合规,不经 Anthropic 认证

切换账号 / 切换方式的正确操作

  1. 换账号:/login 重新走浏览器流程选新账号即可(或 claude auth logout 后重登);claude auth status 随时确认当前身份。
  2. 从 OAuth 切到 API Key:设置 ANTHROPIC_API_KEY 即优先生效——反过来说,想用回订阅,必须先 unset ANTHROPIC_API_KEY(检查 shell 配置文件里有没有残留 export!)。
  3. 企业限制:管控设置里的 forceLoginMethod 可强制 claudeai / console / gateway;此时环境变量凭证会在启动时被直接拦下——"公司电脑用不了 API Key"通常就是它。
认证冲突症状速查"Invalid API key"、"Please run /login"、CLI 好着 Pencil/IDE 却说未登录——九成是多种凭证并存:环境变量 + OAuth + 自定义网关互相打架。排查顺序:env | grep -i anthropic 清掉不要的 → claude auth status → 重新干净登录一次。

七、额度与成本管理(会员必修)

订阅额度是双层水池:滚动窗口额度(约 5 小时一档,用尽等窗口滑动)+ 每周上限(重度使用才会碰到)。具体数值随套餐与官方策略演进,一切以 /usage 面板显示为准。

动作怎么做
看余量/usage:套餐额度、窗口重置时间、按技能/子代理/插件的用量拆解;API 计费用户看 /cost
省额度 · 模型档位任务和档位匹配:硬仗才上高档模型,机械活 /model 降档跑;Auto 模式让系统按任务自动路由,通常比手动全程高档省得多
省额度 · 上下文即成本上下文越大每轮越贵:不相关任务 /clear、长会话及时 /compact、大范围只读摸底交给子代理(隔离上下文,主会话只收结论)、MCP 只装用得上的(工具定义也占上下文)
省额度 · 减少无效轮次先 Plan Mode 审计划再放行执行;两次纠错不成就 /clear 重来——反复拉锯是最贵的用法
超限了怎么办四选一:等窗口重置 / /model 降档继续干 / 临时切 API 计费(设 ANTHROPIC_API_KEY + /config 开关,注意按量计费)/ 升级套餐。"usage limit reached"不是故障,重装重登毫无作用——排错去急救手册 · 计费与额度类

三套值得学的硬核方法论

方法论核心思想怎么落地
Spec 驱动开发把模糊想法先固化为结构化规格,再逐任务执行、每个任务全新上下文"加用户认证"先展开成 research → requirements → design → tasks 四份文档,然后逐项实现;本指南法则五(采访出 SPEC)是它的轻量版;插件化方案见 smart-ralph

复利工程知识跨会话复利:每次干完活,把学到的沉淀为可复用上下文Every.to 的 compound-engineering 插件:/ce-brainstorm → /ce-plan → /ce-work → /ce-review → /ce-compound(写回经验)——"同一个坑,一个月后不再踩第二次";不装插件也能学思想:每次修完疑难杂症让 Claude 把根因写进 CLAUDE.md 或技能

避坑清单

官方点名的五大失败模式 + 社区高频踩坑,对号入座。环境与登录类的坑(凭证冲突、配置不生效、换机翻车)不在此表——请查环境与登录急救手册。

坑症状解法
大杂烩会话任务 A 聊一半插任务 B 再回 A,上下文全是无关信息不相关任务之间 /clear,没有例外
无限纠正循环错→纠正→还错→再纠正,失败尝试把模型往同一坑里带两次未果就 /clear,用吸收教训的新提示重来
CLAUDE.md 过载规则写了一堆,Claude 偏偏无视关键那条无情删减;能转 Hook 的转 Hook,场景化的转 Skill
信任-验证鸿沟产出看着头头是道,边界情况全是窟窿没有验证手段就不算完成;要证据,不要宣称
无边界探索"调查一下 X"没圈范围,读了几百个文件撑爆上下文调研圈紧范围,或者交给子代理
把 Hook 的活交给祈祷"每次都要跑 lint"写进 CLAUDE.md,偶尔就是不跑必须发生的事 = Hook;提示词只是建议
并行会话踩踏两个会话同时改同一个文件,互相覆盖每个并行会话一个 git worktree(claude -w)
把检查点当 git以为 /rewind 能恢复一切,结果 Bash 改的文件回不去检查点只管 Claude 的编辑工具;大节点照常 git commit
评审代理的"必然发现"追着评审报告改,代码越改越防御、越抽象规定只报影响正确性的缺口;其余当可选建议
-p 模式裸奔无人值守脚本花超预算 / 死循环--max-turns + --max-budget-usd + --allowedTools 三件套

③ Hook 模板(拦截敏感写入)

#!/bin/bash  # .claude/scripts/block-sensitive.sh
set -euo pipefail
INPUT="$(cat)"
FILE="$(echo "$INPUT" | jq -r '.tool_input.file_path // empty')"
if echo "$FILE" | grep -qE '\.(env|pem|key)$'; then
  echo "禁止写入敏感文件: $FILE" >&2
  exit 2   # 退出码 2 = 阻断该次工具调用
fi
exit 0

// settings.json 中挂接:
{ "hooks": { "PreToolUse": [
  { "matcher": "Write|Edit",
    "hooks": [{ "type": "command", "command": ".claude/scripts/block-sensitive.sh" }] }
] } }
