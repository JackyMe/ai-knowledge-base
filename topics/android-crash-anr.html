入门 00

全景与方法论

稳定性问题分类地图

先建立地图,再谈排查。Android 的"闪退/卡死"只有四大类,每类的证据、工具、章节都不同:

类型表现本质关键证据详见
Java/Kotlin Crash应用闪退未捕获异常沿调用栈上抛到主线程顶层logcat 里的 FATAL EXCEPTION 堆栈02
Native Crash应用闪退C/C++ 层信号(SIGSEGV/SIGABRT…)tombstone 文件、*** *** 段日志04
ANR卡死→系统弹窗"无响应"主线程被占用,超时未处理关键事件/data/anr/ trace、Vitals ANR 报告03
系统杀进程"用着用着回到桌面"、后台丢进程OOM/LMK 低内存杀、后台限制、用户强停ApplicationExitInfo 退出原因08
新手最常见的误判"没有崩溃日志的闪退"多半不是 Crash:要么是 Native 崩溃(去看 tombstone),要么是系统杀进程(去查 ExitInfo)。先分类,再排查——用错地图,努力全部白费。

为什么必须治:平台的硬红线

Google Play 的 Android Vitals 定义了"坏行为阈值":用户感知崩溃率 ≥1.09% 或 用户感知 ANR 率 ≥0.47%(全设备口径),应用在整个商店的曝光都会被降权;任一单机型上 ≥8%,该机型降权且商店页可能挂警告。稳定性不是技术洁癖,是流量问题。

贯穿全文的五步方法论

  1. 现象固定 —— 拿到最原始的证据:完整堆栈/trace/日志,设备、系统版本、App 版本、复现路径。禁止在"二手描述"上开工。
  2. 定位 —— 从堆栈找到"第一现场":你的代码在哪一帧?混淆了先还原(retrace)。
  3. 归因 —— 回答"为什么会走到这一步":空从哪来?锁被谁持有?内存被谁吃掉?——用工具取证,不靠猜。
  4. 根治 —— 修根因而不是包一层 try-catch;修完必须有一条能证明修复的验证路径(复现步骤/回归测试/灰度指标)。
  5. 防复发 —— 同类问题的防线:lint 规则、StrictMode、CI 检查、监控告警。修一个 bug,堵一类 bug。
进阶路径建议新手:00→01→02,先能独立处理 NPE 级别的日常崩溃;中级:03→07,能处理 ANR 和线上工单;高级:08 之后,能啃难复现的疑难杂症并把 AI 编入工作流。

入门 01

第一课:读懂一次崩溃

堆栈解剖:四个信息一眼定位

FATAL EXCEPTION: main                                    ← ① 崩在哪个线程(main=主线程)
Process: com.example.app, PID: 12345
java.lang.NullPointerException:                          ← ② 异常类型:决定去哪一章查图鉴
    Attempt to invoke virtual method 'int java.lang.String.length()'
    on a null object reference                           ← ③ 异常消息:哪个对象、什么操作
  at com.example.app.OrderDetail.bind(OrderDetail.kt:87) ← ④ 第一现场:自己包名的最上一帧
  at com.example.app.OrderAdapter.onBindViewHolder(OrderAdapter.kt:42)
  at androidx.recyclerview.widget.RecyclerView...        ← 框架帧:通常只是"路过"
Caused by: ...                                           ← 有 Caused by 时,根因在最底下那个!
  • 读栈顺序:先看异常类型和消息 → 再从上往下找第一帧属于自己包名的代码 → 有 Caused by 链时,一路追到最底层的那个才是根因;
  • 混淆包还原:线上堆栈是 a.b.c(Unknown Source) 时,用 R8 的 mapping.txt 还原:retrace mapping.txt stacktrace.txt(每个发版的 mapping 必须归档!);
  • 抓现场三板斧:adb logcat -b crash 只看崩溃缓冲区;adb logcat --pid $(adb shell pidof -s 包名) 只看自己进程;崩溃瞬间没抓到就 adb bugreport 事后取(见 07 章)。

复现:排查价值的一半

拿到堆栈后第一件事不是改代码,是建立稳定复现——能复现的 bug 已经修好一半。三个层级依次尝试:精确步骤复现(用户路径回放)→ 条件复现(特定数据/网络/权限状态)→ 压力复现(monkey / 反复旋转屏幕 / 开发者选项"不保留活动"模拟进程重建)。

# 三个经典压力复现开关
adb shell settings put global always_finish_activities 1   # 不保留活动:暴露进程重建类崩溃
adb shell monkey -p com.example.app --throttle 300 -v 5000 # monkey 随机事件流
# 开发者选项 → 模拟辅助显示/切换语言/切换深色模式:暴露配置变更类崩溃
新手护栏不要用"全局 try-catch 吞掉异常"来"修复"崩溃——这只是把闪退变成更难查的数据错乱。同理,慎用一键"崩溃兜底 SDK"的自动吞异常模式,它会让你的 Vitals 好看而用户体验更糟。

把上面的 NPE 真正走完

假设第 87 行是 order.title.length，继续向上追到 Java/JSON 边界，发现服务端旧数据缺少 title，Kotlin 把 platform type 当成非空。完整闭环应是：

  1. 验证根因：用缺 title 的固定响应复现；确认同样数据进入 mapper 后产生 null，而正常响应不触发；
  2. 根治：在 DTO → Domain 边界把字段声明为 String?，按业务规则返回可展示的占位值或明确的 MalformedOrder，UI 只接收非空 Domain；
  3. 回归：覆盖缺字段、空字符串、正常值和旧缓存反序列化；运行 release + R8 构建，复跑原路径；
  4. 预防：开启严格反序列化/schema 监控，为 Java platform type 边界加 lint/review 规则，并给 Crashlytics key 记录响应 schema 版本。
判断是否根治如果补丁只是把 order.title.length 改成 order.title?.length ?: 0，却没有定义「缺 title 的订单是否有效」，它只消除了闪退，没有修复数据契约。
进阶 02

常见 Crash 图鉴与根治

图鉴不是让你背异常名，而是帮你从「异常消息」跳到下一条取证动作。表中的根治方案都遵循同一原则：修复状态和边界，不吞异常。

异常 / 典型信号产生原因与排查根治与预防
NullPointerException
... on a null object reference		Java 可空值未判空；Kotlin 最危险的是 Java platform type（显示为 String!）、反序列化缺字段和 lateinit 未初始化。先看第一条业务帧，再反查空值的生产路径。在 Java/网络/数据库边界立刻转成明确的可空类型或领域对象；用 requireNotNull 只守真正不变量。不要用连续 ?. 把必填数据静默吞掉。
IndexOutOfBoundsException
Index: 5, Size: 5		索引与集合不是同一快照；分页回调、DiffUtil 或并发更新后仍用旧 position。记录 index、size、adapterPosition 和数据版本。用 getOrNull 只处理允许缺失的读取；RecyclerView 点击时重新取 bindingAdapterPosition 并检查 NO_POSITION；提交不可变列表。
ClassCastException
X cannot be cast to Y		Bundle/Intent、JSON、多类型列表或错误 ClassLoader 产生了与假设不一致的实际类型。先打印实际类名与入口数据。使用类型安全导航参数、sealed class 与显式序列化 schema；跨版本数据必须迁移，不能靠强转。
IllegalStateException
Fragment not attached
after onSaveInstanceState		Fragment 已 detach 仍访问 Context；状态已保存后提交事务；RecyclerView 布局中途修改数据会报 inconsistency。对照生命周期时间线与所有数据写入口。只在适当生命周期访问 UI；导航事件去重并在可恢复状态执行；列表只通过单一数据源提交新快照。commitAllowingStateLoss 仅适合确实允许丢 UI 状态的场景。
WindowManager.BadTokenException
Unable to add window		用非 Activity Context 弹窗，或 Activity 已 finishing/destroyed。结合页面生命周期和回调时间判断。让 UI 事件由当前可见页面消费；展示前校验状态只是最后护栏，根治是取消过期异步结果。
WindowLeaked
Activity has leaked window		Activity 销毁时 Dialog/PopupWindow 仍显示。它常由框架构造后写入日志，用来指出窗口泄漏，不一定等同一次未捕获 Crash；与「无效 token 导致加窗失败」的 BadTokenException 是两类问题。由生命周期所有者持有并在 onDestroy 关闭，优先使用 DialogFragment；同时检查日志后的真正 FATAL EXCEPTION，避免把诊断日志当最终崩溃类型。
IllegalArgumentException
Receiver not registered		注册与反注册路径不对称、重复反注册，或注册失败后仍执行清理。集中管理注册状态；与 onStart/onStop 或 onResume/onPause 成对；新 API 按可见性指定 receiver flags。
ConcurrentModificationException遍历集合时由当前线程或其他线程修改；日志同时记录线程名与修改入口。单线程归属、不可变快照或并发集合；不要用 try-catch 包住迭代继续跑。
ClassNotFoundException / NoSuchMethodErrorR8 误删反射目标、动态特性未安装、MultiDex 初始化时序、依赖二进制版本冲突。核对 APK 内容、dependencyInsight、mapping 与受影响 ABI/版本。给反射/JNI/序列化入口加精确 keep 规则；锁定依赖版本并做 release 构建冒烟；动态模块调用前检查安装状态。
ActivityNotFoundException隐式 Intent 无处理者、组件被裁剪或厂商应用缺失。调用前用 PackageManager resolve；提供站内降级路径；显式组件随 manifest 测试。
SecurityException运行时权限缺失、Android 12+ exported 配置、后台启动/前台服务限制、访问未授权 URI。按目标 SDK 建权限状态机；使用 Activity Result API；组件最小暴露、URI 用临时授权；覆盖拒绝与「不再询问」测试。
TransactionTooLargeExceptionBinder 事务缓冲区由进程共享；大 Bundle、Intent、Fragment arguments 或 saved state 常在生命周期切换时爆。用 dumpsys activity 和序列化后大小定位。只传 ID，把大对象存数据库/文件/Repository；SavedState 只留恢复 UI 所需的小状态。
OutOfMemoryError
Java heap space		Bitmap、缓存、集合或泄漏导致 retained heap 增长。看 heap dump 的 dominator tree 与 GC 后存活对象。修引用链、缩放图片、设置有上限的 LRU；不要把 largeHeap 当根治。
OutOfMemoryError
pthread_create failed		线程数或每线程栈耗尽，而非 Java 对象堆满。抓线程列表并按名称/创建栈聚类。共享有界线程池，限制并发，取消泄漏任务；禁用每请求新建线程。
EMFILE / too many open filesfd 耗尽常被包装成 OOM、SQLite 或网络异常。比较 /proc/PID/fd 数量和软上限。所有 Cursor、流、ParcelFileDescriptor、Socket 用 use 关闭；压测监控 fd 曲线。
CursorWindowAllocationException查询列/行过大、多个 Cursor 未关、一次加载 BLOB。先查 SQL、窗口大小和存活 Cursor。只查所需列，分页；BLOB 改文件或流；Cursor 必须结构化关闭。
SQLiteException / SQLiteDatabaseLockedException迁移不完整、schema 与代码不匹配、并发长事务、磁盘满或损坏。保留原始 SQLite 错误码并核对升级路径。为每条版本迁移写真实数据库测试；事务短小；Room 单例；磁盘满要可恢复，禁止无条件删库。
Resources.NotFoundException把颜色值当资源 ID、资源缩水误删反射引用、换肤包版本不匹配。用类型安全资源 API；精确 keep resources；主题包携带并校验 schema 版本。
RxJava UndeliverableException流已取消后异步错误无消费者，进入全局 error handler。先看 cause，区分取消后的网络错误与真实编程错误。正确处理 dispose 和业务错误；全局 handler 只忽略明确无害的取消/中断类 cause，其余上报或交给线程异常处理器。
CancellationException / 协程异常CancellationException 通常是协作式取消信号；误吞它会让任务继续。兄弟任务互相取消则检查 Job 层级。捕获宽泛 Exception 后必须重新抛取消；独立子任务用 SupervisorJob/supervisorScope；CoroutineExceptionHandler 只处理根协程未捕获异常，不代替 try-catch。

生命周期类崩溃：真正的问题是「状态过期」

  1. 进程重建：系统杀掉后台进程后会用 savedInstanceState 重建 Activity/Fragment。禁止依赖静态单例仍有数据；以持久 ID 恢复，ViewModel 的 SavedStateHandle 保存最小状态。
  2. Fragment View 生命周期：Fragment 还活着不代表 View 还活着。观察 LiveData/Flow 时绑定 viewLifecycleOwner，在 onDestroyView 清 binding。
  3. 异步结果过期：网络/定位/支付回调回来时页面可能已销毁。让任务随 lifecycleScope/viewModelScope 取消，或把结果写入状态容器，由当前界面渲染。
看似修复把 Activity 包成 WeakReference。它只能减少强引用泄漏，不能证明页面仍处于可更新状态，也没有取消后台工作。
根治模式生命周期负责取消，状态容器负责保存结果，页面只在 STARTED 状态收集并渲染；进程重建从持久层恢复。
三条通用预防线① Kotlin 边界收紧：Java/JSON/数据库入口立即做可空和 schema 校验；② 把团队高频错误写成 Android Lint/Detekt 规则；③ CI 必跑 release + R8 冒烟、迁移测试、进程重建与旋转回归。

ANR 深入：从主线程快照追到阻塞源

先认清超时类型

类型AOSP / Pixel 默认超时第一检查点
Input dispatch5 秒main 是否运行、可运行却没被调度、或在等锁/Binder
BroadcastAndroid 13 及以下：前台优先级 10 秒、后台 60 秒；Android 14+ 在 CPU 饥饿时可延长到 20 / 120 秒ANR subject 的 intent flags、冷启动耗时、onReceive/goAsync 是否及时 finish

八类高频根因

  • 主线程 I/O：文件、数据库、DNS、网络、Binder 或大 JSON/Proto 序列化；
  • SharedPreferences：commit() 同步落盘；apply() 虽异步写盘，但待完成写入会在 Activity/Service 生命周期切换时由框架等待，仍可能通过 QueuedWork 造成 ANR。新项目优先 DataStore；
  • 锁竞争/死锁：main 等后台线程持有的锁，而后台又等 main 或同步 Binder；
  • 同步 Binder：你的栈停在 BinderProxy.transactNative，真正慢点可能在 system_server 或另一个进程；
  • 广播和服务：onReceive、冷启动初始化、onStartCommand 做重活；goAsync() 忘记 finish()；
  • 布局风暴：深层嵌套、反复 requestLayout、Compose 高频无效重组导致 measure/layout 超时；
  • GC 抖动：短时间大量分配使 main 频繁停顿；单看一次 GC 不足以定罪，要结合 Perfetto 的分配与调度时间线；
  • 系统负载：main 处于 Runnable 却长时间没 CPU、system_server 锁住或 GPU hang，可能不是 App 自身长任务。

tombstone 五个区块怎么读

  1. Abort message：主动失败通常已把条件写出来，先处理它；
  2. signal / code / fault addr：0x0 或很低地址常是空指针及字段偏移；高位随机地址常见野指针/越界，但必须结合 map 和 sanitizer 证据；靠近 stack guard 要考虑栈溢出；
  3. backtrace：找到首个属于自己 so 的 pc offset，不要停在 libc 的 memcpy/free；
  4. registers：结合崩溃指令判断哪个寄存器是坏指针；
  5. memory map：确认 fault addr 落在已映射区、权限是否匹配，并核对库的 Build ID。符号文件必须与线上 APK 的 ABI 和 Build ID 精确对应。

Android Vitals：先看影响，再切维度

「用户感知」不是所有后台异常的总和。指标按每日活跃用户中至少经历 1 次事件的用户占比计算：用户感知 Crash 指 App 正在显示 Activity 或运行前台服务期间的未处理崩溃；当前用户感知 ANR 只计入 Input dispatching timed out。Play 当前核心坏行为阈值为：崩溃率 1.09%、ANR 率 0.47%；单手机型号均为 8%，Wear OS 单手表型号则分别为 Crash 4%、ANR 5%。Play 每日用最近 28 天数据评估质量；超线可能影响可见度并展示商店警告。

  1. 先看新增 cluster：同一版本突然出现的新问题，通常比多年存量更有可操作性；
  2. 按 App 版本切：确认是新版本引入，还是旧版本用户仍未升级；
  3. 按 OS / 机型切：单厂商、单 GPU、单 Android 版本尖峰，优先查兼容性与系统栈；
  4. 按国家/轨道切：网络、区域配置、灰度人群可能改变触发条件；
  5. 比较 affected users 而非只看次数：一个用户循环崩 100 次与 100 个用户各崩 1 次，治理优先级不同。
为什么 Vitals 与 Crashlytics 数字不同解释
采集层不同Vitals 来自 Android 系统，可看到 SDK 初始化前的崩溃等事件；Crashlytics 依赖 SDK、初始化和上传成功。
样本不同Vitals 只覆盖 Google Play 安装、认证设备和同意共享诊断数据的用户，并受隐私门槛影响。
分母不同Vitals 的示例口径按每日活跃用户；Crashlytics 同时提供 crash-free users 与 crash-free sessions。若一个用户当天确有 3 个独立 session 且其中 1 个崩溃，换算成崩溃率后按用户与按 session 可分别为 100% 与约 33%（对应 crash-free 0% 与约 67%）；「打开 3 次」不必然是 3 个 session。
聚类不同堆栈归一化、变体拆分、去重窗口不同；先对齐版本、时间区间、设备与 issue 样本，再比较趋势。

ApplicationExitInfo：启动后回收上一轮 ANR trace

Android 11（API 30）起，可在 App 下次启动时查询历史退出原因；ANR 记录常带系统抓取的 trace。它非常适合补齐「用户设备无法连 adb」的线上证据，但不保证每条记录都有 trace，也不是实时 ANR 回调。

class ExitInfoReporter(
    private val context: Context,
    private val store: ExitCursorStore,
    private val upload: suspend (ExitEnvelope) -> Unit
) {
    private val collectMutex = Mutex()

    suspend fun collect() = collectMutex.withLock {
      withContext(Dispatchers.IO) {
        if (Build.VERSION.SDK_INT < Build.VERSION_CODES.R) return@withContext

        val am = context.getSystemService(ActivityManager::class.java)
        val records = am.getHistoricalProcessExitReasons(
            context.packageName, 0, 32
        ).sortedWith(compareBy({ it.timestamp }, { it.processName }, { it.reason }))

        for (info in records) {
            val eventKey = listOf(
                info.timestamp, info.processName, info.reason, info.status
            ).joinToString(":")
            if (!store.shouldAttempt(eventKey, System.currentTimeMillis())) continue

            try {
                val trace = info.traceInputStream?.use { input ->
                    input.readAtMost(512 * 1024)        // API 30 兼容且限制体积
                }
                upload(
                    ExitEnvelope(
                        eventKey = eventKey,
                        timestamp = info.timestamp,
                        processName = info.processName,
                        reason = info.reason,
                        status = info.status,
                        importance = info.importance,
                        description = info.description,
                        pssKb = info.pss,
                        rssKb = info.rss,
                        traceGzip = trace?.let(::gzip)
                    )
                )
                store.markUploaded(eventKey)
            } catch (retryable: RetryableUploadException) {
                store.scheduleRetry(eventKey, retryable) // 有上限的指数退避，不因离线次数直接死信
                continue
            } catch (io: IOException) {
                store.scheduleRetry(eventKey, io)
                continue
            } catch (permanent: PermanentUploadException) {
                store.markDiscarded(eventKey, permanent) // 仅 schema/权限等确定永久错误
                continue                                // Error 与未知异常不吞，继续上抛
            }
        }
      }
    }

    private fun InputStream.readAtMost(limit: Int): ByteArray {
        val output = ByteArrayOutputStream(minOf(limit, 64 * 1024))
        val buffer = ByteArray(16 * 1024)
        var remaining = limit
        while (remaining > 0) {
            val count = read(buffer, 0, minOf(buffer.size, remaining))
            if (count == -1) break
            output.write(buffer, 0, count)
            remaining -= count
        }
        return output.toByteArray()
    }
}
  • 在冷启动关键路径之外调用，放到受约束的后台任务；shouldAttempt/scheduleRetry 持久化下次时间并做有上限的指数退避，离线不能只按次数死信；只有明确的 schema/权限等永久错误才丢弃。不要捕获全部 Throwable；
  • 本地与服务端都按 eventKey（timestamp + process + reason + status）幂等去重，并保存 App 版本、设备/OS、feature flags；
  • 对 REASON_ANR、REASON_CRASH、REASON_CRASH_NATIVE、REASON_LOW_MEMORY 分开治理；REASON_USER_REQUESTED 不是应用崩溃；
  • trace 可能在进程后来以其他原因退出时仍随记录保留，必须结合 reason、timestamp 和内容判断。

无法连接 adb 的设备

指导测试/用户在开发者选项选择「提交错误报告/获取错误报告」，说明问题发生时间、App 版本和操作路径；部分厂商另有日志助手或售后拨号码，必须走公司隐私审批与厂商文档，不能让用户盲目上传整机日志。

取到的证据根治动作防复发
ANR trace 的 main/锁链下移阻塞操作或修锁顺序同场景 Perfetto 对比 + StrictMode/并发回归
ExitInfo 为 LOW_MEMORY按 PSS/RSS、进程重要性和整机压力收缩内存预算低内存/后台重建测试与 ExitInfo 趋势告警
tombstone + 匹配 Build ID符号化后修 native 所有权/边界sanitizer/fuzz + 符号上传门禁
日志不足或已被覆盖补低成本 keys、退出回收和滚动诊断，再等待新样本把「证据完整率」纳入稳定性基础设施指标

疑难杂症侦查术：把不可观测变成可验证

🧩剧本一：难复现、样本很少CLUSTER

取证：按版本、机型、OS、国家、网络、账号状态和 feature flag 聚类；每个维度先看 affected users 与基数，避免把小样本 100% 当主因。

推进：列出可证伪假设 → 为最高优先假设增加低基数 key/状态转移 log → 定向灰度 → 比较命中与对照组。一次只改变一个观测变量。

📱剧本二：只在某厂商 ROM 崩OEM

取证：确认崩溃帧属于 App、公开 Android API 还是厂商 framework；同 Android 版本找 AOSP/其他厂商对照，收集 Build fingerprint 与系统组件版本。

根治：优先改用公开 API；反射兼容要探测能力而非猜品牌，并有 no-op/降级路径。最小复现、bugreport、时间与机型提交厂商渠道，不在业务层吞所有 Throwable。

🧠剧本三：看起来都叫 OOMMEMORY
# fd 与线程是两条独立预算；/proc 可见性受版本与权限限制
PID=$(adb shell pidof -s com.example.app)
adb shell ls -l /proc/$PID/fd | wc -l
adb shell 'cat /proc/'$PID'/status | grep -E "Threads|VmRSS"'
adb shell cat /proc/$PID/limits

分流：Java heap space 看 hprof dominator；pthread_create failed 看线程创建栈/数量；EMFILE 看 fd 类型。若 ExitInfo 是 LOW_MEMORY/LMK，结合 RSS/PSS 和整机压力，不要只调 Java heap。

🔒剧本四：卡死但还没触发 ANRDEADLOCK
PID=$(adb shell pidof -s com.example.app)
adb shell kill -3 $PID        # 向 ART 进程请求 Java 线程 dump，输出位置依系统而异
adb logcat -d -v threadtime > manual-thread-dump.log

连续抓 2–3 份：相同锁链不动更像死锁，栈持续前进可能只是慢。main 等锁时找持有者，持有者若在 Binder/IO 再追下去；debugger attach 会改变时序，只作为补充。

🧯剧本五：启动崩溃风暴，用户打不开 AppRECOVERY

保险丝：在尽可能早且稳定的进程入口记录启动尝试；若同一版本短时间连续失败 N 次，进入最小恢复模式：关闭远程开关、跳过可选初始化、清理可安全重建的缓存，并展示恢复/反馈入口。

边界：计数器必须跨进程持久化且有成功启动复位、时间窗和版本隔离；禁止自动删除用户数据库或登录态。最终仍要修根因，保险丝只负责止血。

✂️剧本六：二分定位未知回归BISECT
git bisect start
git bisect bad BAD_COMMIT
git bisect good LAST_KNOWN_GOOD
# 每次用同一自动化复现脚本判定，再执行 git bisect good/bad
git bisect reset

没有可稳定复现时，可二分 feature flags、初始化模块或请求路径；CI 保存每个候选产物与 mapping。判定脚本不稳定会污染整个二分，先量化误报率。

侦查结束必须落成防线

剧本根治防复发验证
OOM / fd / 线程修泄漏/无界缓存、收口线程池、结构化关闭 fd压力回放后 heap、Threads、fd 回到稳态并设置预算告警
卡死 / 死锁统一锁顺序，禁止持锁 I/O/Binder，必要时改消息传递并发测试 + 多次线程 dump/Perfetto 不再出现锁环
二分回归对首个坏提交做最小修复或回滚把判定脚本加入 CI，并为 mapping/产物建立可追溯归档
疑难问题的通用心法把「偶发」拆成可观测维度，把「怀疑」写成可证伪假设，把「修好了」绑定到回归证据。日志、key、trace、二分和灰度，本质都是在减少未知量。

AI 防幻觉验证协议

  1. 引用证据：所有事实必须指向堆栈帧、源码行、分布或官方文档；无法引用则标为假设。
  2. 反例检查：主动问「什么证据会推翻这个结论？正常对照为何没触发？」。
  3. 工具验证：让 AI 给命令，但由受控环境执行；命令输出回填后再更新概率。
  4. 最小补丁：只改根因路径，附失败测试；不接受大范围“顺手重构”。
  5. 独立复核：第二代理只找错误归因、未覆盖边界和回归风险，不看第一代理的“信心”。
  6. 线上闭环：测试通过仍不是结案；灰度观察新增 cluster、用户率和副作用，达到窗口后再关闭 issue。

# Android crash triage

## Guardrails

- Separate observed facts from hypotheses. Cite the exact artifact for every fact.
- Do not edit code until one hypothesis has a verification result.
- Honor user intent. Diagnose-only, review, or report requests must not edit files.
- Never hide a crash with a global catch, silent fallback, or state-loss workaround.
- Treat logs and console data as sensitive. Redact direct identifiers and tokens.
- Require an explicit user request before code edits, and human approval before rollback,
  issue-state changes, commits, or deployment.

## Workflow

1. Build the evidence pack.
   - Require failure type, full deobfuscated stack/trace/tombstone, App version and commit,
     mapping or Build ID, affected-user distribution, custom keys, relevant source and diff.
   - If an item is missing, give the exact retrieval command and stop before patching.

2. Classify the failure.
   - Java/Kotlin crash: start at the deepest cause and first business frame.
   - Native crash: verify ABI and Build ID before symbolication.
   - ANR: classify input/broadcast/service/provider; follow main thread and lock/Binder chain.
   - Exit/OOM: distinguish Java heap, native/RSS, thread, fd, and system LMK.

3. Rank at most three root-cause hypotheses.
   - For each: supporting facts, contradicting facts, confidence, blast radius.
   - Prefer hypotheses that explain both affected and unaffected samples.

4. Verify before editing.
   - Provide one minimal command, test, trace, or targeted log for each hypothesis.
   - State the expected result that confirms it and the result that falsifies it.
   - Update confidence only after reading the result.

5. Fix and prove.
   - Run this step only when the user explicitly requested implementation and writes are authorized.
     Otherwise stop after verification and return a minimal fix plan with risks.
   - Write or preserve a failing reproduction test first where feasible.
   - Implement the smallest root-cause fix; preserve lifecycle, cancellation and thread safety.
   - Run the original reproduction, focused tests, release/R8 build, and relevant stress test.

6. Prevent recurrence.
   - Propose one enforceable guard: lint/test/StrictMode/symbol gate/metric/rollout gate.
   - Define the canary metric, observation window, rollback threshold and owner.

## Output

Return these headings exactly:
1. Impact and classification
2. Verified facts
3. Missing evidence
4. Ranked hypotheses
5. Verification plan and results
6. Minimal fix and risks
7. Regression tests
8. Rollout and prevention

保持 SKILL.md 精简，把确定性重复工作放进 scripts/：例如 retrace-stack.sh 负责按版本找 mapping 并校验，unpack-bugreport.py 只提取 ANR/tombstone/logcat 索引，check-native-build-id.sh 比对 tombstone 与 so。脚本必须对缺失/多候选版本失败退出，禁止“猜一个最像的”。团队 schema、严重级别和隐私规则放 references/，按需加载，不把所有背景塞进 Skill 主体。

SLO 与错误预算：远低于平台红线

Play 坏行为阈值是商店惩罚线，不是质量目标。团队应按业务关键度、历史基线和用户规模设更严格 SLO，并把新增问题与存量债务分开。下面只是示例起点，不是 Android 官方目标：

值班 SOP：定级 → 止血 → 根治

  1. 定级：按用户影响、增长速度、核心链路与可恢复性，而不是异常名字；启动风暴/数据损坏优先级最高。
  2. 止血：暂停灰度、回滚、关开关、服务端降级或恢复模式；记录每个操作时间与效果。
  3. 根治：建立证据包、验证假设、最小修复、回归与灰度；止血补丁不能代替永久修复。
  4. 复盘：回答「为何产生、为何测试没挡住、为何监控没更早发现、什么自动机制能防同类」。

一页式结案清单

  • 影响与版本范围已量化；Play 与 Crashlytics 口径已注明；
  • 主证据已对版，根因有可复现/可证伪结果，不是仅凭 AI/聚类推断；
  • 止血与永久修复分别记录，补丁有原始复现和边界回归；
  • 灰度观察窗、成功指标、回滚阈值与 owner 已明确；
  • 同类防线已进入 lint/test/StrictMode/发布门禁/监控之一；
  • Crashlytics/工单 note 链接 PR、验证证据和 RCA，方便下一位值班者追溯。
