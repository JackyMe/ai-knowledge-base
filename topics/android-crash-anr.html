入门 00

全景与方法论

稳定性问题分类地图

先建立地图,再谈排查。Android 的"闪退/卡死"只有四大类,每类的证据、工具、章节都不同:

类型表现本质关键证据详见
Java/Kotlin Crash应用闪退未捕获异常沿调用栈上抛到主线程顶层logcat 里的 FATAL EXCEPTION 堆栈02
Native Crash应用闪退C/C++ 层信号(SIGSEGV/SIGABRT…)tombstone 文件、*** *** 段日志04
ANR卡死→系统弹窗"无响应"主线程被占用,超时未处理关键事件/data/anr/ trace、Vitals ANR 报告03
系统杀进程"用着用着回到桌面"、后台丢进程OOM/LMK 低内存杀、后台限制、用户强停ApplicationExitInfo 退出原因08
新手最常见的误判"没有崩溃日志的闪退"多半不是 Crash:要么是 Native 崩溃(去看 tombstone),要么是系统杀进程(去查 ExitInfo)。先分类,再排查——用错地图,努力全部白费。

为什么必须治:平台的硬红线

Google Play 的 Android Vitals 定义了"坏行为阈值":用户感知崩溃率 ≥1.09% 或 用户感知 ANR 率 ≥0.47%(全设备口径),应用在整个商店的曝光都会被降权;任一单机型上 ≥8%,该机型降权且商店页可能挂警告。稳定性不是技术洁癖,是流量问题。

贯穿全文的五步方法论

  1. 现象固定 —— 拿到最原始的证据:完整堆栈/trace/日志,设备、系统版本、App 版本、复现路径。禁止在"二手描述"上开工。
  2. 定位 —— 从堆栈找到"第一现场":你的代码在哪一帧?混淆了先还原(retrace)。
  3. 归因 —— 回答"为什么会走到这一步":空从哪来?锁被谁持有?内存被谁吃掉?——用工具取证,不靠猜。
  4. 根治 —— 修根因而不是包一层 try-catch;修完必须有一条能证明修复的验证路径(复现步骤/回归测试/灰度指标)。
  5. 防复发 —— 同类问题的防线:lint 规则、StrictMode、CI 检查、监控告警。修一个 bug,堵一类 bug。
进阶路径建议新手:00→01→02,先能独立处理 NPE 级别的日常崩溃;中级:03→07,能处理 ANR 和线上工单;高级:08 之后,能啃难复现的疑难杂症并把 AI 编入工作流。
入门 01

第一课:读懂一次崩溃

堆栈解剖:四个信息一眼定位

FATAL EXCEPTION: main                                    ← ① 崩在哪个线程(main=主线程)
Process: com.example.app, PID: 12345
java.lang.NullPointerException:                          ← ② 异常类型:决定去哪一章查图鉴
    Attempt to invoke virtual method 'int java.lang.String.length()'
    on a null object reference                           ← ③ 异常消息:哪个对象、什么操作
  at com.example.app.OrderDetail.bind(OrderDetail.kt:87) ← ④ 第一现场:自己包名的最上一帧
  at com.example.app.OrderAdapter.onBindViewHolder(OrderAdapter.kt:42)
  at androidx.recyclerview.widget.RecyclerView...        ← 框架帧:通常只是"路过"
Caused by: ...                                           ← 有 Caused by 时,根因在最底下那个!
  • 读栈顺序:先看异常类型和消息 → 再从上往下找第一帧属于自己包名的代码 → 有 Caused by 链时,一路追到最底层的那个才是根因;
  • 混淆包还原:线上堆栈是 a.b.c(Unknown Source) 时,用 R8 的 mapping.txt 还原:retrace mapping.txt stacktrace.txt(每个发版的 mapping 必须归档!);
  • 抓现场三板斧:adb logcat -b crash 只看崩溃缓冲区;adb logcat --pid $(adb shell pidof -s 包名) 只看自己进程;崩溃瞬间没抓到就 adb bugreport 事后取(见 07 章)。

复现:排查价值的一半

拿到堆栈后第一件事不是改代码,是建立稳定复现——能复现的 bug 已经修好一半。三个层级依次尝试:精确步骤复现(用户路径回放)→ 条件复现(特定数据/网络/权限状态)→ 压力复现(monkey / 反复旋转屏幕 / 开发者选项"不保留活动"模拟进程重建)。

# 三个经典压力复现开关
adb shell settings put global always_finish_activities 1   # 不保留活动:暴露进程重建类崩溃
adb shell monkey -p com.example.app --throttle 300 -v 5000 # monkey 随机事件流
# 开发者选项 → 模拟辅助显示/切换语言/切换深色模式:暴露配置变更类崩溃
新手护栏不要用"全局 try-catch 吞掉异常"来"修复"崩溃——这只是把闪退变成更难查的数据错乱。同理,慎用一键"崩溃兜底 SDK"的自动吞异常模式,它会让你的 Vitals 好看而用户体验更糟。
