定调 01

范式与角色升级:为什么 12 周敢立军令状

过去转全栈要 1-2 年,慢在"招式":语法、API、样板代码、配置。2025-2026 的范式两级跳(Vibe Coding → 智能体工程)把招式成本砍掉了一个数量级——AI 承包语法与样板,人负责架构与验收。

维度传统(Pre-AI)Vibe Coding(~2025)智能体工程(2026)

质量保障断点调试跑起来看看TDD + 类型系统 + 自动化测试拦截
可达规模无上限但慢Demo/单页多租户 SaaS、商业化产品

系统编排者:你的新岗位说明书

职责谁干你的具体动作
需求拆解 / 架构决策 / 验收标准你写清 Goal + Context + Constraints + Done-when,再让 AI 动手

范式跳变对你个人的意义它抹平的是招式差距(你缺的),抹不平的是内功差距(你有的)。内功厚、招式旧的人,恰好是这个窗口期的最大受益者。方法论出处与研究依据见心法页 · 编程范式。
硬核迁移 02

语言层:Kotlin → TypeScript 速通

好消息:Kotlin 和 TypeScript 是"表亲"——都是带类型推断、空安全、函数式风格的现代语言。这一章把你 90% 的 Kotlin 语感直接兑换成 TS,并标出剩下 10% 真正要重学的部分。用法:先通读一遍,之后当字典查。

① 数据建模:data class → type + 展开语法

▍Kotlin
data class User(
  val id: Long,
  val name: String,
  val email: String? = null
)

val u2 = u.copy(name = "新名字")
▍TypeScript
type User = {
  id: number;        // 没有 Long,数值统一 number
  name: string;
  email?: string;    // ? = 可缺省(≈ 默认 null)
};

const u2 = { ...u, name: "新名字" }; // copy() ≈ 展开

② 空安全:你已经会了,换符号而已

▍Kotlin
val name: String? = user?.profile?.name
val shown = name ?: "匿名"
val len = name!!.length  // 危险的断言
▍TypeScript(strict 模式)
const name: string | null = user?.profile?.name ?? null;
const shown = name ?? "匿名";   // ?: → ??
const len = name!.length;       // !! → !,同样危险
前提:tsconfig 必须开 strict"strict": true(含 strictNullChecks)是 TS 空安全的总开关——不开等于回到 Java 的 NPE 时代。新项目脚手架默认开着,永远不要为了消红线关掉它;同理,写 any = 把类型系统整个让出去,AI 生成的代码里见到 any 就要求它改成具体类型或 unknown。

③ 密封类 → 判别联合(TS 最重要的一课)

▍Kotlin
sealed interface UiState
object Loading : UiState
data class Success(val data: List<Item>) : UiState
data class Error(val msg: String) : UiState

when (state) {
  Loading    -> showSpinner()
  is Success -> render(state.data)
  is Error   -> toast(state.msg)
}  // 编译器保证穷尽
▍TypeScript
type UiState =
  | { kind: "loading" }
  | { kind: "success"; data: Item[] }
  | { kind: "error"; msg: string };

switch (state.kind) {
  case "loading": showSpinner(); break;
  case "success": render(state.data); break; // 自动收窄
  case "error":   toast(state.msg); break;
  default: { const _x: never = state; } // 穷尽检查
}

判别联合(discriminated union)是 TS 建模的心脏:接口返回、表单状态、事件类型全用它。never 兜底那行 = 编译期穷尽检查,漏一个分支直接标红——和 sealed + when 完全同一心智。

④ 协程 → async/await:语法更简单,语义要小心

▍Kotlin
viewModelScope.launch {
  try {
    val user = repo.fetchUser()  // suspend
    _state.value = Success(user)
  } catch (e: Exception) {
    _state.value = Error(e.message)
  }
}
▍TypeScript
async function loadUser() {
  try {
    const user = await fetchUser(); // 返回 Promise
    setState({ kind: "success", user });
  } catch (e) {
    setState({ kind: "error", msg: String(e) });
  }
}

四个语义差异,记住能少踩一半坑:

  • Promise 创建即执行,不像协程可以冷启动;"先建后跑"要包一层函数;
  • 没有结构化并发与作用域取消:没有 viewModelScope 帮你收尸,取消要手动(AbortController 或"存活标记");
  • 并行等待:Promise.all([a, b]) ≈ awaitAll(async{a}, async{b});
  • 单线程,无数据竞争:不需要 Mutex/synchronized——但也意味着 CPU 重活会阻塞一切(04 章细讲)。

⑤ 集合操作:几乎白送

KotlinTypeScript备注
list.map { it.name }list.map(x => x.name)it → 显式参数
list.filter { it.age > 18 }list.filter(x => x.age > 18)同款
list.firstOrNull { … }list.find(…)(返回 undefined)null → undefined
list.sumOf { it.price }list.reduce((s, x) => s + x.price, 0)reduce 万能但啰嗦
mapOf("a" to 1){ a: 1 } 或 new Map()普通对象当轻量 map 用
解构 val (a, b) = pairconst [a, b] = arr; const { name } = user对象解构比 Kotlin 强

⑥ 真正的新东西:结构化类型(鸭子类型)

▍Kotlin:名义类型
class Cat(val name: String)
class Dog(val name: String)

fun feed(cat: Cat) { … }
feed(Dog("旺财"))  // ❌ 编译错误:类型不同
▍TypeScript:看结构不看名字
type Cat = { name: string };
type Dog = { name: string };

function feed(c: Cat) { … }
feed({ name: "旺财" } as Dog); // ✅ 结构一致即兼容

影响:接口对得上就能传,灵活但"防串门"要靠判别字段(见③)。前两周会别扭,之后你会发现它是 AI 协作的润滑剂——AI 生成的对象只要形状对就能用。

本章验收(能答上再往下走)① strict 模式下 string | null 为什么必须处理 null 分支?② 判别联合怎么做穷尽检查?③ Promise 和协程在"取消"上的本质差异?——答案都在上面,答不出就再读一遍。系统学习(教材/学法/练习全配好):学习包①。
硬核迁移 03

UI 层:Compose → React,你已经会了 70%

写过 Compose 的人学 React,不是学新范式,是换方言:组件即函数、状态驱动 UI、单向数据流、副作用受控——四大心智完全同构。逐条对照:

① 组件与状态:remember → useState

▍Compose
@Composable
fun Counter() {
  var count by remember { mutableStateOf(0) }
  Button(onClick = { count++ }) {
    Text("点击 $count 次")
  }
}
▍React
function Counter() {
  const [count, setCount] = useState(0);
  return (
    <button onClick={() => setCount(count + 1)}>
      点击 {count} 次
    </button>
  );
}
React 第一杀手坑:状态必须不可变更新Compose 里 list.add(item) 配 mutableStateListOf 能触发重组;React 里直接改对象/数组不会触发重渲染——必须造新引用:setItems([...items, item])、setUser({ ...user, name: "新" })。转型头一个月 80% 的"UI 怎么不刷新"都是它。

② 副作用:LaunchedEffect → useEffect

▍Compose
LaunchedEffect(userId) {   // key 变化时重启
  profile = repo.load(userId)
}

DisposableEffect(Unit) {
  val listener = attach()
  onDispose { listener.detach() }
}
▍React
useEffect(() => {
  let alive = true;               // 手动"结构化并发"
  load(userId).then(p => { if (alive) setProfile(p); });
  return () => { alive = false; }; // 清理 ≈ onDispose
}, [userId]);                      // 依赖数组 ≈ key

依赖数组三形态:[a, b] 变化才跑(≈ LaunchedEffect(a, b))、[] 只挂载跑一次、不写 = 每次渲染都跑(几乎总是 bug)。返回的函数在"重跑前 + 卸载时"执行,对标 onDispose。

③ 其余概念一表兑换

ComposeReact备注
重组 recomposition重渲染 re-render同款心智:状态变 → 函数重跑 → 差异更新
状态提升(hoisting)状态提升(lifting state up)连名字都一样:子组件收 value + onChange
CompositionLocalContext(createContext/useContext)跨层传值,少用,全局配置才用
LazyColumn { items(list) }list.map(x => <Row key={x.id} />)key 必须稳定唯一,别用 index(diff 依据)
derivedStateOfuseMemo(() => …, [deps])昂贵派生值才用;React Compiler 时代大部分可不写
Modifier.padding(16.dp)…className="p-4 …"(Tailwind)都是链式原子修饰,心智同构
Slot API(content lambda)children prop组件组合方式一致
预览 @Preview浏览器热更新 + Storybook(可选)Web 的反馈环更快

④ 布局:ConstraintLayout 思维 → Flex/Grid 思维

▍Compose
Row(
  horizontalArrangement = Arrangement.SpaceBetween,
  verticalAlignment = Alignment.CenterVertically,
  modifier = Modifier.fillMaxWidth().padding(16.dp)
) {
  Text(title, fontWeight = FontWeight.Bold)
  Badge(count)
}
▍Tailwind
<div className="flex w-full items-center
                justify-between p-4">
  <span className="font-bold">{title}</span>
  <Badge count={count} />
</div>

// Row ≈ flex(默认横向), Column ≈ flex flex-col
// 响应式:md:flex-row 表示"中屏以上才横排"

组件库直接用 shadcn/ui:不是依赖包而是把源码拷进你项目的组件集——相当于 Material 组件给你完整源码随便改,配 Tailwind 是 2026 的社区默认款。

硬核迁移 04

架构层:从 App 架构到 Web 全栈架构

这是全路线唯一没有 Android 对应物、必须从零建立的心智:你的代码第一次同时跑在两个地方——服务器和浏览器。搞懂"哪段代码在哪跑、能碰到什么",App Router 就通了 80%。

① 服务端组件 vs 客户端组件(App Router 第一课)

▍服务端组件(默认):只在服务器跑
// app/projects/page.tsx
import { db } from "@/lib/db";

export default async function ProjectsPage() {
  // 直接 async 查库!没有 ViewModel/Repository/DTO 长链
  const projects = await db.query.projects.findMany();
  return <ProjectList projects={projects} />;
}
// 产物是 HTML:代码与密钥不下发浏览器,天生 SEO
▍客户端组件:要交互才声明
// components/like-button.tsx
"use client";  // 声明:这个组件下发到浏览器跑
import { useState } from "react";

export function LikeButton() {
  const [liked, setLiked] = useState(false);
  return (
    <button onClick={() => setLiked(!liked)}>
      {liked ? "❤️" : "🤍"}
    </button>
  );
}
判断题服务端组件客户端组件("use client")
用 useState/useEffect/onClick?❌ 不能✅ 必须它
直接查数据库/读密钥?✅ 专属能力❌ 绝对禁止
默认选哪个?默认服务端;确实需要交互的叶子组件才 "use client"——交互下沉到叶子,是 App Router 的组件划分第一原则

② 架构映射:MVVM 的每一块去哪了

Android(MVVM/Clean)Next.js 全栈说明
Activity/Fragment + NavGraphapp/ 目录文件树(文件即路由)app/projects/[id]/page.tsx = 详情页,目录即导航图
ViewModel + StateFlow服务端组件(读)+ Server Actions(写)+ 少量客户端 state大部分"状态管理"消失了:数据在服务端现取现渲染
Repository + Retrofit/RoomDrizzle ORM 直查 PostgreSQL没有"客户端缓存数据库"这一层,缓存交给框架(下表)
Interceptor(鉴权)middleware.ts(会话检查/重定向)+ RLS 在数据库层兜底(05 章)
深链 Deep LinkURL 本身Web 的 URL 天生即深链,还免费带 SEO
Application 全局初始化root layout(app/layout.tsx)全局壳:字体、Provider、导航栏

③ 渲染与缓存:SSR/SSG/ISR 三分钟版

模式HTML 何时生成用在哪类比
SSG 静态生成构建时一次性落地页、博客、文档——SEO 主力打包时烘焙好的资源
ISR 增量再生构建时 + 每 N 秒后台刷新商品页、内容站(revalidate: 3600)带 TTL 的缓存
SSR 请求时渲染每次请求登录后的个性化页面(dashboard)每次现算的接口

水合(Hydration)一句话:服务器发来的 HTML 先展示(快、可收录),浏览器随后加载 JS 给它"通电"变可交互——首屏快与可交互之间的桥。客户端组件越少,水合越便宜,这就是"交互下沉到叶子"的性能理由。

④ 事件循环:把 ANR 直觉带过来

▍Android 心智
// 主线程做 IO/重计算 → ANR
// 解法:切线程
withContext(Dispatchers.IO) {
  heavyWork()
}
▍Node/浏览器心智
// 单线程事件循环:IO 天生异步不阻塞 ✅
// 但同步重计算会卡死整个循环 = 全局 ANR
// 解法:不是切线程,是"切执行位置"
//   浏览器端 → Web Worker
//   服务端   → 异步任务队列(Trigger.dev)
本章验收① 一个带点赞按钮的文章页,哪些组件放服务端、哪些放客户端?② ISR 的 revalidate 与 Room 缓存策略像在哪?③ 服务端组件里能不能写 onClick,为什么?——第 4-5 周卡壳时回来重读本章,这是全路线最陡也最值钱的一段坡。
硬核迁移 05

数据层:Room/Retrofit → Supabase 全家桶

心智转变一句话:数据库从"App 的本地缓存"升级为"产品的唯一真相源",而且安全规则直接写在数据库里。这一章包含全手册最重要的安全内容(RLS),一人公司没有安全团队,数据库层的默认拒绝就是你的安全团队。

① SQL 速成:给用惯 Room 的你

-- 一个 SaaS 的最小三表(直接在 Supabase SQL Editor 跑)
create table profiles (          -- 用户资料(auth.users 由 Supabase 管)
  id uuid primary key references auth.users(id),
  display_name text not null,
  created_at timestamptz default now()
);

create table projects (
  id uuid primary key default gen_random_uuid(),
  owner_id uuid not null references auth.users(id),
  title text not null,
  status text not null default 'active',   -- 判别联合的数据库版
  created_at timestamptz default now()
);
create index on projects (owner_id);       -- 按外键查询必建索引

-- Room 的 @Query 变成了真 SQL:
select p.title, count(t.id) as task_count
from projects p left join tasks t on t.project_id = p.id
where p.owner_id = $1
group by p.id order by p.created_at desc;

与 Room 的换算:@Entity → create table;@Dao 方法 → SQL/ORM 查询;Migration → 迁移文件(Drizzle 自动生成);TypeConverter → 不需要,PostgreSQL 类型比 SQLite 富得多(jsonb/数组/全文检索都原生)。

② RLS:全手册最重要的一节

-- Row Level Security:权限规则写进数据库,任何入口都绕不过
alter table projects enable row level security;
-- ⚠️ 开启后默认拒绝一切访问,必须显式开门——这正是它的价值

create policy "只能读自己的项目"
  on projects for select
  using ( owner_id = (select auth.uid()) );

create policy "只能建属于自己的项目"
  on projects for insert
  with check ( owner_id = (select auth.uid()) );

create policy "只能改/删自己的项目"
  on projects for update
  using ( owner_id = (select auth.uid()) );
对比Android 习惯:代码层校验RLS:数据层强制
规则在哪散在各接口/Interceptor 里集中在表上,一处声明处处生效
漏写一处= 数据泄露= 请求被拒(默认拒绝,fail-safe)
AI 写错代码可能直接把库查穿最多查到自己的数据
两把钥匙,一条人命红线anon key(公开钥匙,受 RLS 管)可以出现在浏览器;service_role key(万能钥匙,无视 RLS)只允许存在于服务端环境变量——放进任何 NEXT_PUBLIC_ 开头的变量或客户端代码 = 全库裸奔。每张表建完第一件事:开 RLS、写策略、用两个测试账号互查验证。

③ Drizzle ORM:类型安全的 @Dao

▍Schema 定义(≈ @Entity)
// db/schema.ts
export const projects = pgTable("projects", {
  id: uuid("id").primaryKey().defaultRandom(),
  ownerId: uuid("owner_id").notNull(),
  title: text("title").notNull(),
  status: text("status").notNull().default("active"),
  createdAt: timestamp("created_at",
    { withTimezone: true }).defaultNow(),
});
// npx drizzle-kit generate → 自动产出迁移 SQL
▍查询(≈ @Dao,全程类型推断)
const list = await db
  .select()
  .from(projects)
  .where(eq(projects.ownerId, user.id))
  .orderBy(desc(projects.createdAt));
// list 的类型自动推断,字段拼错编译期即报

await db.insert(projects)
  .values({ ownerId: user.id, title });

④ Server Actions:Retrofit 的整个存在理由消失了

// app/projects/actions.ts —— 完整的"写操作"标准姿势
"use server";
import { z } from "zod";
import { revalidatePath } from "next/cache";

const CreateProject = z.object({
  title: z.string().min(1, "标题必填").max(80, "太长"),
});

export async function createProject(formData: FormData) {
  // 1. 服务端再验一次身份(永远不信客户端)
  const user = await getUser();
  if (!user) return { error: "请先登录" };

  // 2. Zod 校验入参 —— 运行时的类型守卫
  const parsed = CreateProject.safeParse({ title: formData.get("title") });
  if (!parsed.success) return { error: parsed.error.issues[0].message };

  // 3. 落库(RLS 在数据库层再兜一道底)
  await db.insert(projects).values({ ownerId: user.id, title: parsed.data.title });

  // 4. 让列表页缓存失效 → UI 自动更新
  revalidatePath("/projects");
}
// 页面里:<form action={createProject}> 直接绑,零 API 路由、零 DTO、零 Retrofit

三道防线记心里:客户端校验(体验)→ Zod 服务端校验(入口)→ RLS(兜底)。AI 生成的任何写操作,照这个模板审。

学习包:六大体系逐个击破(学什么 · 怎么学 · 怎么实操)

大体系不能靠一句"去看官方文档"打发。每个学习包四件套:主教材(精确到章节)→ 学法 → 实操任务 → 过关线。总原则:一个体系只跟一个主线资源,做完为止——同时开三门课是教程地狱的正门。

name: CI
on: [pull_request]

③ 环境变量:三层纪律

层放什么纪律
本地 .env.local开发用密钥永远在 .gitignore;泄露过的密钥一律轮换
Vercel Preview测试环境密钥(Stripe 测试 key 等)与生产隔离,PR 环境不许碰真钱真数据

④ 上线安全清单(第 8 周与每次大版本前过一遍)

  1. RLS 全覆盖:每张表都开启且有策略;用双账号互查实测,不是看代码"应该没问题";
  2. 钥匙边界:service_role 只在服务端;全局搜一遍 NEXT_PUBLIC_ 确认无密钥;
  3. 入口全校验:每个 Server Action / API 路由都有 Zod + 身份检查(05 章模板);
  4. Webhook 验签:Stripe 回调必须验证签名——不验签 = 任何人可伪造"已付款";
  5. 限流:登录/发信等敏感端点加速率限制(Upstash Ratelimit 半小时接入);
  6. 监控:Sentry 接好前后端错误上报——你的"崩溃率看板";
  7. 备份:确认 Supabase 自动备份开启,并亲手演练过一次恢复;
  8. 依赖audit:CI 里加 npm audit --audit-level=high,高危不上线。
发布纪律(从你的 Android 直觉搬过来)灰度=preview 链接先发小群;回滚=Vercel 一键回上个部署(比商店审核回滚幸福一万倍);监控告警响了先回滚再排查——老兵都懂,别因为发版太容易就丢了纪律。

# 项目约定 —— Next.js + Supabase SaaS

## 技术栈
Next.js 16 App Router · TypeScript strict · Tailwind + shadcn/ui
Drizzle ORM · Supabase(Auth/PG/RLS) · Stripe · Vercel

## 硬性规则
- 默认服务端组件;仅叶子交互组件加 "use client"
- 禁止 any;优先判别联合建模状态
- 所有写操作走 Server Action:身份检查 + Zod 校验 + revalidatePath
- 每张新表:同时给出建表 SQL、RLS 策略、Drizzle schema、回滚语句
- service_role 密钥只允许出现在服务端;发现即报错
- 改动完成后运行:npm run lint && npx tsc --noEmit && npm test

## 沟通约定
- 我是 10 年 Android 背景:新概念用 Compose/Kotlin/Room 类比解释
- 先给计划再动手;方案超过 3 个文件的改动必须先列清单

一人公司商业闭环:技术之外的另一半

① 选题:五问过滤器(动手前必答)

问题及格线
1. 我自己是不是这个问题的用户?是——自用产品的迭代直觉碾压猜需求
2. 目标用户有没有付费习惯?开发者工具/效率工具/B 端小工具 > 面向大众的免费习惯区
3. 搜索得到吗?存在明确搜索词(如 "xxx generator/converter")——SEO 才有抓手
4. 一个人 12 周内做得出 MVP 吗?砍到剩一个核心动作还成立,才算
5. 头部垄断了吗?有 3-5 个活着的小竞品 = 市场验证过且容得下小玩家(完全空白反而危险)

② 获客:SEO 是你选 Web 全栈的全部理由,用足它

  • 产品即内容:每个功能页/教程页/对比页都是搜索入口,SSG 天生可收录;
  • 程序化 SEO:用数据批量生成长尾页("X vs Y"矩阵、模板库)——AI + Next.js 的组合技,一人也能做出千页站;
  • 冷启动三渠道:build in public(独立开发者社区/即刻)、答题式获客(在别人提问处解决真问题顺带露出)、转型复盘本身就是内容(你的 12 周就是连载素材)。

③ 收款:国内主体的现实问题(多数教程避而不谈)

方案适用现实提醒
Stripe海外用户对主体所在地有要求,国内个人直接开通有门槛;常见路径是海外主体,先算清成本
MoR(Paddle / Lemon Squeezy 等)海外收款想省事Merchant of Record 替你处理税务合规,抽成略高——一人公司早期非常值
国内通道(微信/支付宝)国内用户企业资质基本是硬门槛;决定做国内市场前先查清这条
默认建议一人公司常见最优解:英文产品 + 海外市场 + MoR 收款——市场大、客单价高、合规路径最短。平台政策变化快,动手前以官方最新说明为准。

④ 从 0 到 1000 美元 MRR:里程碑行动表

里程碑主要矛盾行动重点
$0 → 第一笔"有人要吗"上线前:落地页+等待列表验证(50 个邮箱 > 5000 行代码);首发:等待列表+社区+限时早鸟;目标是 10 个付费者的真实反馈,不是收入
→ $100 MRR"续订吗"盯三个数:访客→注册、注册→付费、月流失;每周和 2 个用户真人对话;砍掉没人用的功能
→ $1000 MRR"可复制吗"把有效获客渠道加倍投入;程序化 SEO 上量;涨价实验(涨价比加功能常常更有效);其余指标此前都是噪音

定价心法:订阅制起步价别定太低(涨价难降价易)、给年付折扣、三档定价造心理锚点。
