Pencil × Claude Code
最佳实践全景指南

Pencil 是住进 IDE 的 AI 设计画布,Claude Code 是终端里的智能编码代理,MCP 把两者连成一条「设计上下文零损耗」的流水线。本指南聚合官方文档、社区教程、实战博客与开源技能,整理出一套可直接落地的完整打法。

导读 00

为什么是 Pencil + Claude Code

它解决的核心问题:设计-代码鸿沟

传统流程里,设计师在 Figma 精雕细琢,开发者对着标注重写一遍——间距差一点、字重不对、hover 态没文档,最终上线的东西「像设计稿的表亲」。这不是谁的错,而是设计与开发两套工具链、两个事实来源的结构性问题。

通用 AI 编码工具也没能根治:它们看不到你的设计系统、品牌色和间距阶梯,只能基于全网见过的模式生成「看起来还行」但不属于你产品的组件。

这对组合的答案Pencil 把设计放进代码仓库(.pen 文件),Claude Code 通过 MCP 直接读取画布的精确坐标、令牌与结构来生成代码——不是看图猜,而是读数据。设计与代码同仓库、同上下文、同一次提交。社区把这种体验称为「vibe design」:自然语言生成 UI → Git 直接管理 → 一句话转成代码。

能力版图

角色提供什么
Pencil无限画布、组件/变量/插槽体系、Figma 导入、内置 AI 聊天、.pen 文件格式、无头 CLI
Claude Code代理式编码:多文件读写、执行命令、计划模式、技能(Skills)与 MCP 扩展
MCP桥梁协议:让 Claude Code 获得 batch_design、batch_get、get_screenshot、snapshot_layout、get_variables 等设计操作工具

上手 01

环境搭建标准流程

综合官方文档与 DevelopersIO 实测教程,这是社区验证过的最稳妥路径(以 VS Code 为例,Cursor 同理):

  1. 安装 Claude Code CLI 并认证:npm install -g @anthropic-ai/claude-code-cli,运行 claude 走浏览器登录。
  2. 安装 Pencil 扩展:VS Code 扩展面板搜索 "Pencil",认准发布者标识 highagency.pencildev。
  3. 启动 Pencil:点击状态栏左下角的 Pencil 铅笔图标(不出现就重启 VS Code)。
  4. 邮箱激活:首次启动输入邮箱 → 收验证码 → 填码登录。
  5. 验证 MCP 连接:启动 Claude Code,执行 /mcp,确认 pencil 显示为 connected。MCP 配置在安装时已自动写好,无需手工配置。
  6. 创建 design.pen:放在项目工作区根目录,与代码同仓库——这是后续一切「AI 同时看到设计与代码」玩法的前提。

自动写入 .claude.json 的 MCP 配置长这样(路径因平台而异):

"mcpServers": {
  "pencil": {
    "command": ".../highagency.pencildev-x.y.z/out/mcp-server-darwin-arm64",
    "args": ["--app", "visual_studio_code"],
    "type": "stdio"
  }
}
新项目接入的推荐姿势(官方)先建 design.pen → 在 IDE 打开 → 激活 → 保持 .pen 与代码同仓库。首次可右键画布 → Open Welcome File 跟着内置练习走一遍。
三个高频坑① 一切 AI 功能的前提是 claude 已登录——大多数「连不上」都出在这;② 暂无自动保存,勤按 Cmd/Ctrl + S;③ 用 Codex 的用户注意 config.toml 可能被改动,先备份。

上手 02

四种核心工作流

社区实践收敛出四种可复用的工作流范式,可按项目阶段自由组合:

WORKFLOW A
自然语言原型(Vibe Design)

在 Claude Code 里用几行中文/英文描述,直接生成整页设计;再到画布上手工微调。DevelopersIO 实测:旅行相册、面包店点单页、咖啡店仪表盘、社区公告板、员工门户五种页面,均达到「可直接当原型用」的质量。

描述需求 → AI 生成 .pen → 画布微调 → Git 提交
WORKFLOW B
设计 → 代码

画布定稿后,一句 "Generate a Next.js page from this design" 让 Claude Code 读取精确结构与令牌生成代码。提示词中写明技术栈(React/Vue/Tailwind/shadcn-ui…)效果最好。

画布设计 → Cmd/Ctrl+K → 指定技术栈生成 → 审查合并
WORKFLOW C
代码 → 设计

让 AI 把 src/components 里的既有组件在画布上重建("Recreate the Button component from src/components/Button.tsx"),结构、布局、样式一并导入,老项目也能补出设计稿。

读取组件源码 → 画布重建 → 视觉改进 → 回写代码
WORKFLOW D
双向同步 + Git 单一事实源

变量层打通 CSS ↔ Pencil("Create Pencil variables from my globals.css"),.pen 与代码一起提交、一起回滚——设计与实现之间「零漂移」。改设计的 PR 里能直接 diff 设计变更。

变量同步 → 设计/代码并行迭代 → 同一 commit 提交

实战 04

设计系统工程化

单页生成只是起点;社区里最成熟的打法,是把 Pencil + Claude Code 当成设计系统的生产与消费流水线。

五阶段方法论(社区 Claude Code Skill)

社区流传较广的 /create-design-system 技能把「从想法到设计系统」拆成五个必须逐一确认的阶段,核心思想是每个阶段边界都停下来给人确认,绝不一口气冲到底:

阶段做什么关键动作
① 想法挖掘把模糊想法问清楚:给谁用、核心工作流是什么、MVP 边界在哪锁定平台(Web/移动)、气质(专业/活泼/极简)、5–7 个关键屏幕;主动抵制范围蔓延
② 风格方向探索视觉方向用风格标签生成 2–3 组方向;鼓励用户丢 Dribbble/Behance/竞品截图,真实视觉参考胜过千言万语
③ 品牌方向板在 .pen 里做 2–3 块方向板供用户挑选每块含:色板(主/辅/强调/中性/语义色)、字阶(H1–H4/正文/说明)、样例组件、3–5 个气质关键词;A 稳妥、B 大胆、C 出格
④ 组件库按定稿方向构建可复用组件库先设变量(设计令牌),再建必备组件(按钮三态、输入框、卡片、导航、徽章),全部标记 reusable;截图给用户确认
⑤ 关键屏幕用组件库拼装第一阶段定下的屏幕从最重要的屏幕开始定调;组件以 ref 实例插入;永远用真实感内容,拒绝 lorem ipsum;逐屏截图确认
该技能的通用守则(同样适用于日常使用)不停展示工作(每个大步骤后截图);阶段间必须确认;宁可「无聊但易读」不要「新奇但难懂」;给节点起好名字;画布保持分区整洁。

企业级案例:Lullabot 品牌库

Lullabot 工程师 Jerad Bitner 的做法展示了「库文件」的正确姿势:把整个品牌系统装进一个可导入的设计库——56 个颜色变量、Circular + IBM Plex Mono 的字体令牌、4px 网格的间距阶梯、16 个可复用组件(按钮、卡片、表单输入、导航链接、徽章、标签、引用块)。

  • 公司内部工具 OpptyNet 导入它,Upstraker 导入它,下一个新工具还导入它——第二个项目起,视觉语言就不用从零搭。
  • 他的原话概括:「有了共享设计库,你不再纠结用哪种蓝,而开始思考界面真正要做什么。」
对应的 Pencil 机制把品牌文件转成 .lib.pen 设计库(图层面板 → Libraries → Turn this file into a library),即可被任意项目导入;库内组件更新会同步到所有使用处。注意:转库操作不可逆。

来源:TobyScr / create-design-system skill · Jerad Bitner: Designing with Pencil and Claude Code · 官方 Design Libraries

实战 05

高效提示词模式

具体性法则

❌ 低效"弄好看点" / "加个表单" / "做个页面"
✅ 高效"按钮内边距加到 16px,颜色改为蓝色" / "登录表单:邮箱、密码、记住我、提交按钮" / "三要素:页面类型+内容清单+配色气质"

迭代金字塔(官方推荐节奏)

  1. 先框架 — "创建仪表盘布局"
  2. 再细化 — "加带导航项的侧边栏"
  3. 后打磨 — "导航项加 hover 态"
  4. 终收尾 — "间距对齐 8px 网格"

社区共识与此一致:抵制「一个巨型提示词生成完整应用」的诱惑。小块设计 → 生成 → 审查 → 精修 → 下一块,质量远高于一次性梭哈。

上下文注入技巧

  • 引用既有资产:"用我们现有的按钮组件"、"遵循变量里的间距阶梯"、"和 Header 组件风格保持一致"——让 AI 复用而不是重造。
  • 点名设计系统:"按 Material Design 原则"、"用 shadcn/ui 风格"、"遵循 design-system.pen 里的规范"。
  • 画布选区即上下文:在画布上选中的元素会自动进入 AI 上下文;话题切换时点 "New Agent" 清空上下文重开。
  • 要求先解释再动手:对大改动,让 AI 先说明打算怎么改,确认后再执行,配合 Claude Code 的计划模式(Plan Mode)效果更佳——计划错了一句话就能纠正,改完了才发现错,代价是几百行 diff。

验证闭环

  1. 画布上目视检查;
  2. 图层面板核对结构;
  3. 让 AI 调 get_screenshot 截图自检复杂布局;
  4. snapshot_layout 排查重叠与错位。

进阶 06

高级玩法

Pencil CLI:无头设计与批量生产

Pencil CLI 与桌面应用共用同一编辑引擎,却完全不需要 GUI——这让「设计」变成可脚本化的构建步骤:

# 一句话生成设计文件
pencil --out login.pen --prompt "Create a modern login page with email, password, social login"

# 修改既有设计并导出 2 倍图
pencil --in dashboard.pen --out dashboard-v2.pen --prompt "Add a sidebar navigation"
pencil --in design.pen --export hero.png --export-scale 2

# 批量任务(每个任务独立编辑器实例,顺序执行)
pencil --tasks batch.json

# 简单任务换便宜模型,控制成本
pencil --out simple.pen --model claude-haiku-4-5 --prompt "Create a simple 404 page"

接入 CI/CD

在组织设置的 Developer Keys 里创建 CLI Key,流水线中即可自动跑设计任务:

export PENCIL_CLI_KEY=pencil_cli_...
export ANTHROPIC_API_KEY=sk-ant-...
pencil --out onboarding.pen --prompt "Create a 3-step onboarding flow"

配合 pencil interactive 交互 shell,还能在脚本里直接调 MCP 工具做精细操作(batch_design 建节点 → get_screenshot 出图 → save() 落盘)。

Script 节点:数据驱动的设计

对图表、网格、纹样这类「手搭很痛苦」的结构,用 Script 节点让代码生成图层:声明 @input 参数即可在属性面板获得可视化旋钮;把输入绑到变量(如 $primary)还能自动响应主题切换。官方示例仓库 highagency/pencil-scripts 提供现成的图表/网格/纹样脚本。

让 AI 直接写脚本
Create a script that renders a horizontal bar chart with 5 bars. Add inputs for the bar color, gap between bars, and an array of values.

与 Claude Code 生态组合

  • Skills(技能):把重复流程固化成技能,如社区的 /create-design-system(五阶段设计系统)和 pencil-to-code(.pen → React 组件的规范化转换)。
  • Plan Mode:大功能先规划后动手;规划阶段只读不写,方向错了一句话就能掉头。
  • 设计令牌文件:让 Claude Code 维护一份 design-tokens.ts / design-tokens.css,作为设计与代码共同的单一事实源,项目演进时一处更新、两边生效。
  • 版本控制兜底:代理式修改多文件时偶尔会弄坏东西——频繁 commit 给自己留检查点;.pen 的撤销/重做有限,Git 就是你的后悔药。

进阶 07

避坑清单

方法论层面

最大的坑:先画像素,后想目的Jerad Bitner 复盘 OpptyNet 项目时的核心教训——一上来就让 AI 出 UI,产出「看着挺像回事」却是在没打地基的房子上刷漆。正确顺序:先定义目的(给谁用?成功长什么样?例如「业务同事要在 60 秒内从邮件摘要做出 go/no-go 判断」)→ 内容建模 → 设计令牌 → 然后才是像素。他发现的关键规律:缺筛选器、AI 功能被藏进标签页,这些是「目的问题」,再多视觉打磨也修不好。

操作层面

坑规避方法
元素散落画布,AI 读不到永远在命名的 Frame 里设计——MCP 工具按 Frame 读取数据
图层叫 "Group 47" / "Rectangle 12"语义化命名(如 PrimaryActionButton、NotificationBadge),名字会随 MCP 传给 AI,帮它理解意图而不只是外观
视觉之外的信息 AI 不知道交互行为、断点、数据要求、无障碍要求——写进注释/说明,AI 无法推断它看不到的东西
一个巨型提示词生成全站小步迭代:设计一小块 → 生成 → 审查 → 精修 → 下一块
忘保存丢工作无自动保存;勤按 Cmd/Ctrl + S + 大改动前 Git 提交
撤销撤不回去撤销/重做比常规设计工具受限,靠 Git 历史回滚
手改 .pen 文件改坏官方明确:不要手动编辑 .pen,走编辑器或 MCP 工具
Figma 图片粘贴失败已知限制:复制粘贴不带图片,改用完整文件导入或单独导图
假文占位骗过了自己用真实感内容评估设计,lorem ipsum 会掩盖信息层级问题
多人实时协作预期落空暂无 multiplayer,用 Git 分支 + PR 评审设计

认知层面

  • AI 产出质量 ∝ 你给的上下文质量——这是全部实践浓缩成的一句话。模糊提示 + 无上下文 = 通用模板;具体提示 + 画布/风格/仓库上下文 = 属于你产品的代码。
  • Pencil 不是 Figma 替代品(至少现在不是):功能没有 Figma 全,强项是内部工具、快速原型与设计-代码一体化场景;重协作、重高保真交付的团队仍需权衡。
  • 工具迭代快:菜单位置、命令语法都可能变;掌握原理(上下文经由结构化协议流向代码代理)比记住界面更长久。

进阶 08

真实案例研究

案例一:OpptyNet — 七个版本,三小时,一条对话线

Lullabot 内部的政府招标(RFP)筛选工具,全程用 Pencil + Claude Code 对话式设计:

版本方案结论
V1卡片列表 + 筛选栏 + 分页好看但视觉太重,扫 20–50 条太慢
V2密集表格 + 行内评分条扫得快但太冷,「比不过她已有的表格」

V6按设计评审修正:统计压成单行、加筛选药丸、Ask AI 固定底部常驻解决的是「目的问题」
