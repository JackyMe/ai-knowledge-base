方法论 00

环境问题的本质:一张地图

Claude Code 与 Codex 的"环境问题"看似五花八门,拆开只有四类根源。先认清根源,后面所有章节都是在这张地图上定位、修复。

根源典型表现为什么会发生
① 多个安装并存版本忽新忽旧、更新了没变化、命令行为诡异native 安装器、npm 全局、Homebrew/WinGet、IDE 扩展自带副本——四种渠道可以同时存在,PATH 里谁靠前谁说了算
② 凭证多源打架"明明登录了却报 Invalid API key"、会员却走 API 计费、organization disabled环境变量里的 key 优先级高于你浏览器登录的订阅凭证——一条别人留下的 export ANTHROPIC_API_KEY 就能劫持你的整个会话
③ 配置层叠覆盖"改了不生效"、公司电脑某些设置锁死企业管控 > 命令行参数 > 项目本地 > 项目 > 用户层——你改的层可能被更高层悄悄覆盖
④ 网络与系统环境登录卡住、TLS 报错、OAuth 回调失败、下载 403公司代理/私有 CA/防火墙拦截回调端口;系统时钟不准还会让 token 校验失败
一句话心法遇到任何环境问题,不要上来就重装。先用下一章的五层诊断法定位问题在哪一层——90% 的"疑难杂症"是第②类(凭证打架),清一个环境变量就好,重装反而引入第①类问题。

方法论 01

五层诊断法 + 一键体检脚本

系统性检查方法只有一条纪律:按层从下往上查,查完一层再查下一层。每层都有明确的检查命令和"正常长什么样"。

L1
二进制层 —— 跑的到底是哪个程序?

which -a claude / which -a codex 列出 PATH 里的所有同名命令;claude --version 确认版本。正常:只有一条路径。异常:多条路径 → 多安装并存,先做减法。

L2
环境变量层 —— 有没有变量在劫持行为?

env | grep -iE 'anthropic|claude|codex|openai|proxy'。正常:一片空白(纯订阅用户)。异常:任何一条 ANTHROPIC_API_KEY / ANTHROPIC_BASE_URL / *_PROXY 都值得盘问来历。

L3
凭证层 —— 当前用的是谁的什么凭证?

Claude Code 会话内 /status 看 Login method(有 API key 行说明环境变量在接管);Codex 用 codex login status。确认账号邮箱、组织是不是你自己的。

L4
配置层 —— 哪一层配置在生效?

Claude Code:claude doctor 体检、/config 看生效值来源;Codex:/debug-config 打印配置层叠。改了不生效 = 被更高层覆盖或该键需要重启会话。

L5
网络层 —— 请求真的出得去、回得来吗?

curl -sI https://downloads.claude.ai/claude-code-releases/latest 测连通;公司网查代理变量与私有 CA;OAuth 卡住查回调端口(Codex 是 1455)。

为什么是这个顺序下层决定上层:程序都跑错了(L1),看凭证没意义;环境变量(L2)会篡改凭证选择(L3);凭证对了配置(L4)才谈得上;前四层都对还不通,才轮到网络(L5)。逆序排查会白忙。

一键体检脚本(只读,复制即用)

把五层检查合成一个脚本。只读不改,在任何机器上都可以放心跑——上手陌生电脑前先跑一遍,30 秒看清全部状态:

#!/bin/bash
# ai-env-check.sh —— Claude Code / Codex 环境一键体检(只读)
# 用法:bash ai-env-check.sh   (macOS / Linux / WSL)
echo "═══ L1 二进制层 ═══"
for c in claude codex; do
  echo "--- $c ---"
  if command -v "$c" >/dev/null 2>&1; then
    which -a "$c"; "$c" --version 2>/dev/null | head -1
  else echo "(未安装或不在 PATH)"; fi
done
[ -d ~/.claude/local ] && echo "⚠️  发现 legacy npm 安装:~/.claude/local(建议删除)"
npm -g ls @anthropic-ai/claude-code 2>/dev/null | grep -q claude-code && echo "⚠️  发现 npm 全局安装(与 native 安装并存会打架)"

echo; echo "═══ L2 环境变量层 ═══"
env | grep -iE 'anthropic|claude|codex|openai|https?_proxy|no_proxy|node_extra_ca' \
  || echo "(干净:无相关环境变量)"
echo "--- shell 配置文件里的残留(重点检查别人留下的 export)---"
grep -nE 'ANTHROPIC|CLAUDE_|CODEX|OPENAI' ~/.zshrc ~/.zprofile ~/.bashrc ~/.bash_profile ~/.profile 2>/dev/null \
  || echo "(干净:rc 文件无相关配置)"

echo; echo "═══ L3 凭证层 ═══"
codex login status 2>/dev/null || echo "codex:未登录或未安装"
[ -f ~/.codex/auth.json ] && echo "codex:存在 auth.json(明文凭证)"
echo "claude:进入会话运行 /status 查看 Login method 与账号归属"

echo; echo "═══ L4 配置层(文件清单)═══"
ls -la ~/.claude/settings.json ~/.claude.json ~/.codex/config.toml 2>/dev/null
ls -la .claude/settings.json .claude/settings.local.json .mcp.json 2>/dev/null \
  || echo "(当前目录无项目级配置)"

echo; echo "═══ L5 网络层 ═══"
curl -sI --max-time 8 https://downloads.claude.ai/claude-code-releases/latest | head -1
curl -sI --max-time 8 https://auth.openai.com | head -1
echo "体检完成 ✅  逐层核对:多安装?可疑变量?账号归属?配置来源?网络连通?"
Windows 用户PowerShell 下对应命令:where.exe claude 查二进制;Get-ChildItem Env: | Where-Object Name -match 'ANTHROPIC|CODEX|OPENAI' 查环境变量;别忘了查"系统属性 → 环境变量"里的用户级变量和 $PROFILE 文件——图形界面设的变量不在 rc 文件里。

检查项依据:官方 Troubleshoot installation and login · 官方 Authentication(2026-07 版)

Claude Code 02

Claude Code · 凭证机制与优先级

Claude Code 的登录问题,八成源于一个被忽视的事实:它同时支持 7 种凭证来源,且有严格的优先顺序。你以为在用订阅,它可能在用一个你根本不知道存在的环境变量。

凭证优先级(官方完整顺序,从高到低)

  1. 云厂商通道:设置了 CLAUDE_CODE_USE_BEDROCK / CLAUDE_CODE_USE_VERTEX / CLAUDE_CODE_USE_FOUNDRY 时,走 AWS/GCP/Azure 凭证(注:已登录的企业网关 gateway 会话比这还高)
  2. ANTHROPIC_AUTH_TOKEN:以 Bearer 头发送——LLM 网关/中转站方案都靠它
  3. ANTHROPIC_API_KEY:以 X-Api-Key 头发送。交互模式首次会弹一次"是否使用该 key"的确认;非交互模式(-p)下只要存在就直接用,不问你
  4. apiKeyHelper 脚本:settings.json 里配置的动态取密钥脚本(默认 5 分钟或收到 401 时刷新)
  5. CLAUDE_CODE_OAUTH_TOKEN:claude setup-token 生成的一年期 token(CI 用)
  6. 订阅 OAuth:/login 浏览器登录的凭证——排在最后。这就是为什么任何一条环境变量都能"劫持"你的订阅
会员必读:为什么我在付两份钱?你是 Pro/Max 会员,但机器上残留一条 ANTHROPIC_API_KEY(自己以前实验用的、前同事留下的、教程里抄的)——它优先于订阅生效,你的每次对话都在按 API 计费,或者因为那个 key 所属组织已停用而报 This organization has been disabled。解法:unset ANTHROPIC_API_KEY + 从 shell 配置文件删掉 export 行 + /status 确认 Login method 回到订阅。

凭证存在哪(官方口径)

平台位置说明
macOS系统钥匙串(Keychain),加密钥匙串锁定或密码不同步会导致登录保存失败——见急救表
Linux~/.claude/.credentials.json(权限 0600)设了 CLAUDE_CONFIG_DIR 则在该目录下
Windows%USERPROFILE%\.claude\.credentials.json继承用户目录访问控制

凭证由 /login / /logout 全权管理,不要手动编辑或跨机拷贝。登录有效期临近 5 天时启动会提示 Your login expires in N days,跑一次 /login 即续期(v2.1.203+);过期后每次请求报 Login expired · Please run /login。

三个必会状态命令

命令告诉你什么
/status(会话内)当前生效凭证:Login method 行显示订阅账号;若出现 API key 行,说明环境变量正在接管;登录过期会显示 Expired
claude doctor(终端)全面体检:安装方式、版本、配置 schema 错误、macOS 钥匙串访问
claude --safe-mode裸配置启动——怀疑自定义配置搞坏一切时,用它做对照实验

来源:官方 Authentication(含 Authentication precedence) · 官方 Settings(2026-07 版)。配置层叠与 settings 文件详解见Claude Code 指南 · 配置与账号全解。

Codex 04

Codex · 凭证机制与存储

2026 年中文档搬家提醒Codex 官方文档已从 developers.openai.com/codex 迁往 learn.chatgpt.com/docs(旧链接 308 重定向,仍可达)。搜教程时注意甄别新旧域名与时效。

三种登录方式

方式怎么用要点
ChatGPT OAuth(默认)codex login → 浏览器,回调 localhost:1455订阅额度;Fast mode 等功能仅此方式可用;走 ChatGPT 工作区权限与数据政策
API Key平台后台取 key 传给 codex login按量计费;部分依赖云端的功能受限;官方推荐 CI 用
设备码codex login --device-auth无浏览器环境首选;部分工作区需管理员先启用,否则无头机登录会被卡住

凭证存哪、怎么续期

  • 存储三选一:config.toml 里 cli_auth_credentials_store = "file" | "keyring" | "auto"。file = 明文 ~/.codex/auth.json;keyring = 系统钥匙串;auto 优先钥匙串。
  • 自动续期:token 过期(约 8 天不用会变 stale)时,Codex 运行中自动刷新并回写 auth.json——CI 持久化方案就建立在这个机制上(见第 08 章)。
  • CLI 与 IDE 扩展共享同一份凭证:一端 logout,另一端也会掉线——"IDE 突然要求登录"多半是你在终端登出过。
  • 多账号:官方暂无一等公民支持(上游 issue #4432 讨论中);可用 CODEX_HOME 隔离目录实现(剧本见第 06 章),社区还有 codex-multi-auth 等管理工具。
  • 企业管控:forced_login_method / forced_chatgpt_workspace_id 可锁定登录方式与工作区,凭证不匹配会被直接登出——公司机器"换不了登录方式"是策略使然。
auth.json = 你的账号本体明文 token,当密码对待:不 commit、不贴聊天、不进截图。需要跨机复用优先走设备码登录,实在要拷贝也只通过安全通道,用完即删。

来源:官方 Authentication(现 learn.chatgpt.com/docs/auth)· 官方 Config Reference(2026-07 版)。config.toml 层叠与 Profile 详解见 Codex 指南 · 配置与账号全解。

场景剧本 06

剧本 A · 共用电脑 / 接手"别人配置过"的电脑

最高频的翻车现场:你是 Claude 会员,坐到一台别人用过的电脑前,一顿操作后各种报错。根因几乎总是同一个:上一位使用者的凭证与环境变量还活着,而它们的优先级比你的登录高。

上机三查(动手配置之前先做)

  1. 查环境变量与 rc 残留:跑第 01 章体检脚本,重点看 L2 输出——别人 rc 文件里的 export ANTHROPIC_API_KEY / ANTHROPIC_BASE_URL(中转站配置)/ OPENAI_API_KEY 就是定时炸弹。
  2. 查凭证归属:claude 进会话 /status 看登录的是谁的账号;codex login status 同理。是别人的就先 /logout / codex logout。
  3. 查全局配置污染:~/.claude/CLAUDE.md、~/.claude/settings.json、~/.codex/config.toml、~/.codex/AGENTS.md 里是不是别人的偏好与指令——这些会直接影响你的每次对话行为。

按共用程度选方案

长期共用一台机器各开各的系统账号(最正规)macOS/Windows 各建用户:home 目录天然隔离,~/.claude、钥匙串、~/.codex 互不可见,零冲突零泄密

# Claude Code:CLAUDE_CONFIG_DIR 决定"老家"在哪

场景剧本 08

剧本 C · SSH / 容器 / 无头机 / CI

没有浏览器的环境,登录是另一套打法。两个工具的官方推荐路径不同,别互相套用。

Claude Code:三条路径

  1. SSH/WSL2/容器交互使用:正常跑 claude,把终端里的 OAuth URL 复制到本地浏览器完成登录,浏览器给出的 code 粘回终端(粘贴失灵就用 claude auth login 从 stdin 读)。
  2. CI / 脚本(订阅额度):在本地跑 claude setup-token 生成一年期 OAuth token,存进 CI Secret,运行环境设 CLAUDE_CODE_OAUTH_TOKEN。注意 --bare 模式不读它,bare 场景改用 ANTHROPIC_API_KEY 或 apiKeyHelper。
  3. CI(API 计费):直接 ANTHROPIC_API_KEY 环境变量——最简单,但走按量计费。

Codex:官方三方案按优先级

  1. 设备码(首选):codex login --device-auth,本地任意浏览器输一次性码。工作区可能需要管理员先启用设备码权限。
  2. 拷贝 auth.json(官方兜底):有浏览器的机器登录后把 ~/.codex/auth.json 安全复制到目标机(scp / docker cp)。若本机凭证在钥匙串而非文件,先把 cli_auth_credentials_store 设为 file 重登一次。
  3. SSH 端口转发:ssh -L 1455:localhost:1455 user@remote 后在远端 codex login,本地浏览器走标准流程。

Codex CI 长期认证(官方 auth.json 续期循环)

ChatGPT 订阅额度跑 CI 的官方姿势——利用 Codex 运行时自动刷新 token 并回写 auth.json 的机制:

  1. 可信机器上 codex login(file 存储),把生成的 auth.json 存入 CI 的 Secret 管理;
  2. 每次任务:恢复 auth.json → 跑 Codex(token 约 8 天不刷会失效,运行中自动续期)→ 回写刷新后的 auth.json 到 Secret;
  3. 红线:不要多个并发任务共享同一份 auth.json;不要每次都用最初的旧文件覆盖新文件;刷新链断了就回可信机器重新登录播种。自托管 runner 直接给个持久 CODEX_HOME 更省事。

终极方案 09

万能修复:三级重置方案

确实排查不动了,就分级重置。纪律:从 L1 开始逐级升级,每级之后验证,好了就停——直接上核弹会把"能定位问题的现场"一起炸掉,而且多数问题 L1/L2 就能解决。

L1 软重置(2 分钟)—— 解决 70% 的问题

# 清掉当前 shell 的劫持变量(只影响本会话,零风险)
unset ANTHROPIC_API_KEY ANTHROPIC_AUTH_TOKEN ANTHROPIC_BASE_URL
unset OPENAI_API_KEY

# 干净重登
claude   # 会话内:/logout → 退出 → 重新 claude → /login → /status 验证
codex logout && codex login && codex login status

L2 凭证与残留重置(5 分钟)—— 再解决 20%

# 1. 根除 rc 文件里的残留 export(编辑器打开逐行删,别 sed 盲删)
grep -nE 'ANTHROPIC|CLAUDE_|CODEX|OPENAI' ~/.zshrc ~/.zprofile ~/.bashrc ~/.profile

# 2. 清凭证存储后重登
#    Claude Code:/logout 已清;macOS 疑难时解锁钥匙串:
security unlock-keychain ~/Library/Keychains/login.keychain-db
#    Codex:登出并删凭证文件
codex logout; rm -f ~/.codex/auth.json

# 3. 校准系统时钟(token 校验依赖它)——macOS:
sudo sntp -sS time.apple.com

# 4. 重新登录 + 验证
claude   # /login → /status
codex login && codex login status

L3 核弹重置(15 分钟)—— 备份 → 清干净 → 官方渠道重装

# ═══ 第一步:备份核心资产(手写配置是你的,其他都可再生)═══
mkdir -p ~/ai-cli-backup
cp -r ~/.claude/settings.json ~/.claude/CLAUDE.md ~/.claude/skills ~/.claude/agents ~/ai-cli-backup/ 2>/dev/null
cp -r ~/.codex/config.toml ~/.codex/AGENTS.md ~/ai-cli-backup/ 2>/dev/null

# ═══ 第二步:卸掉所有并存安装(Claude Code)═══
npm uninstall -g @anthropic-ai/claude-code 2>/dev/null
rm -rf ~/.claude/local                        # legacy npm 安装
brew uninstall --cask claude-code 2>/dev/null # 若曾用 brew 装过

# ═══ 第三步:清状态目录(已备份,放心删)═══
rm -rf ~/.claude ~/.claude.json
rm -rf ~/.codex

# ═══ 第四步:官方渠道各装一份 ═══
curl -fsSL https://claude.ai/install.sh | bash
npm install -g @openai/codex

# ═══ 第五步:恢复配置 + 重新登录 + 全面验证 ═══
cp ~/ai-cli-backup/settings.json ~/ai-cli-backup/CLAUDE.md ~/.claude/ 2>/dev/null
cp -r ~/ai-cli-backup/skills ~/ai-cli-backup/agents ~/.claude/ 2>/dev/null
mkdir -p ~/.codex && cp ~/ai-cli-backup/config.toml ~/ai-cli-backup/AGENTS.md ~/.codex/ 2>/dev/null
claude doctor && claude    # /login → /status 确认订阅生效
codex login && codex login status
核弹前的三个确认① ~/.claude/projects/ 里有没有还想留的会话转录(删了就没了);② 备份目录里的文件真实存在(ls ~/ai-cli-backup 亲眼确认);③ Windows 用户对应清理 %USERPROFILE%\.claude 与 %USERPROFILE%\.codex,并检查用户环境变量面板。
重置完还不行?那基本可以确定不在你机器上:① 官方服务状态(status.anthropic.com / status.openai.com);② 订阅状态(claude.ai/settings / ChatGPT 设置);③ 企业管控策略(问 IT);④ 带着 claude doctor 输出 / codex-login.log 去提 issue 或找官方支持——这是唯一需要"等别人"的情形。
