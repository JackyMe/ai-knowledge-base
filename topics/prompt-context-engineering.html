全景:从提示词工程到上下文工程

2023 年大家谈"提示词工程"(怎么问一句话);2026 年官方与社区的重心已经移到上下文工程(怎么管理模型看到的一切)。原因很简单:当 AI 从"回答一个问题"进化成"自主干完一个任务",决定成败的不再是那一句问话,而是整个上下文窗口里装了什么——系统提示、工具定义、历史消息、检索进来的文件,全都是"提示词"。

提示词工程(单轮)上下文工程(智能体时代)
优化对象一条指令怎么写整个上下文窗口装什么、何时装、何时扔
核心约束表达是否清晰注意力预算有限:塞得越多,单位 token 的关注度越低
典型动作加示例、给格式校准系统提示高度、精简工具集、压缩/笔记/子代理
官方定义找到最小的高信号 token 集合,最大化期望结果的概率(Anthropic 上下文工程文章)
这一页和心法页的分工心法页回答"AI 能干什么、该怎么用"(场景与原则);本页回答"具体怎么说、怎么喂"(技法与工程)。心法页的七条心法之一"上下文是第一资源"——本页就是那句话的展开说明书。

技法 01

基础七技法(每个配 ❌/✅ 对照)

全部出自官方提示词工程文档,按"日常收益从高到低"排序。前两个技法解决 80% 的质量问题。

技法① 清晰直接:把模型当"能力超强但不知道背景的新同事"

❌ 模糊帮我优化一下这段文案 ——优化什么?给谁看?什么算好?模型只能猜,猜错了你还得返工。
✅ 具体把这段产品介绍压缩到 100 字以内,保留"免费试用 30 天"和价格信息,语气改为面向企业采购决策人的正式风格,删掉所有感叹号。

自检口诀:目标、受众、约束、成功标准,四样齐了再发送。写完问自己一句:"一个不了解背景的新同事,拿到这段话能一次做对吗?"

技法② 示例 few-shot:一个好例子胜过三段描述

❌ 纯描述帮我把用户反馈分类,要考虑情感倾向和问题类型… ——形容词堆得再多,模型对"你想要的样子"仍是模糊的。
✅ 给样例按以下格式分类用户反馈: 输入:"充了会员还有广告,退钱!" 输出:{"情感":"负面","类型":"计费投诉","紧急":true} 输入:"深色模式什么时候上?" 输出:{"情感":"中性","类型":"功能请求","紧急":false} 现在处理:"..."

2026 版官方要点:示例要多样且典范(canonical)——挑 2-5 个能代表不同情形的标准样本,不要把一堆边缘 case 全塞进去(那是在教模型钻牛角尖)。

技法③ 思维链:先想后答

❌ 直接要答案这两个方案哪个好?
✅ 要求推理过程先分别列出两个方案在成本、风险、可维护性上的优劣,再给结论和理由。把分析放在 <thinking> 里,结论放在 <answer> 里。

适用:数学、逻辑、多步决策、代码分析。不适用:简单事实问答(徒增延迟)。推理模型时代它依然有效——显式的分析框架引导模型"往哪个方向想"。

技法④ 结构标签:长材料与指令分舱

❌ 混装总结这份合同,重点看违约条款,合同如下:甲方…(三千字)…另外帮我注意付款周期 ——指令埋进材料里,模型可能漏掉"付款周期"。
✅ 分舱<contract> …三千字合同原文… </contract> <task> 总结上述合同,重点:1) 违约条款 2) 付款周期 </task>

用 XML 风格标签(<doc>、<example>、<task>)把不同性质的内容围起来。长上下文额外规律:关键指令放开头或结尾,长材料放中间——首尾位置的注意力最高。

技法⑤ 角色设定:一句话换专家视角

❌ 万能角色迷信你是世界第一的天才全能专家大师… ——堆砌头衔不增加信息量,见 07 章反模式。
✅ 具体到岗位与立场你是审阅初创公司融资协议的律师,立场是保护创始人。逐条指出对创始人不利的条款并给修改建议。

角色的价值不在"变聪明",在于圈定知识域、口径与立场——同一份合同,买方律师和卖方律师看出的问题不同。

技法⑥ 预填充:替它起个头

❌ 任其发挥输出总带一段"好的,我来帮你…"的开场白,格式时对时错。
✅ 预填开头API 里直接把 assistant 回复的开头写成 { ——模型会顺着补完 JSON,跳过一切废话;或预填 方案A: 锁定输出结构。

对话界面里的等效技巧:在指令末尾写"直接以表格开始输出,不要任何开场白"。

技法⑦ 任务分解:一个大提示词不如一条流水线

❌ 巨型提示词读这 10 篇文章,提炼观点,对比分歧,写成报告,再配 5 条推文… ——步骤越多,每步质量越不可控,错误还会向后传染。
✅ 链式分解第 1 轮:逐篇提炼观点(可并行);第 2 轮:对比分歧;第 3 轮:成稿;第 4 轮:衍生推文。每步产出可检查,错了只重跑那一步。

来源:Anthropic 官方提示词工程文档(全系列) · 2026 版最佳实践

技法 02

结构化输出与格式控制

要让 AI 的产出能被程序消费(脚本解析、表格导入、自动化流水线),"差不多的 JSON"等于零分。三层手段按可靠性递增:

层做法可靠性
① 提示词约束给出字段清单 + 一个完整输出示例 + "只输出 JSON,不要解释"够日常;偶发多话/漏字段
② 预填充assistant 开头预填 {(API)或"直接以 { 开始"基本消灭开场白问题
③ Schema 强约束API 的结构化输出 / JSON schema 能力(如 Claude 的 output format、工具强制调用),把格式交给系统层保证生产级;写自动化必用
模板 · 提示词层结构化输出
从下面的会议记录中提取行动项,输出 JSON 数组,每项字段: - task: string,动词开头 - owner: string,负责人名;没提到则 "未指定" - due: string,YYYY-MM-DD;没提到则 null 不确定的信息宁可标 null,不要编造。只输出 JSON。 <notes>…会议记录…</notes>
两个魔法句式① "不确定的宁可标 null / 说不知道,不要编造"——官方 2026 最佳实践确认:显式允许表达不确定性能显著降低幻觉;② "先输出你对任务的理解,我确认后再执行"——高风险任务的一道保险闸。

进阶 03

上下文工程核心:注意力是稀缺资源

原则① 注意力预算:多塞≠更好

上下文窗口不是硬盘,是注意力的分母:塞进的每个 token 都在稀释模型对其他 token 的关注。上下文过长时模型出现"迷失中间"(long-context degradation)——中段信息被忽略、早先指令被"忘记"。所以官方对好上下文的定义是一道优化题:最小的高信号 token 集。

  • 与任务无关的材料,再"可能有用"也不放;
  • 能引用路径就不粘全文,能摘要就不给原文;
  • 会话跑偏或换任务:新开会话比"在废墟上继续盖"便宜得多(对应 CC 的 /clear 纪律)。

原则② 高度校准:系统提示的"金发姑娘区"

写系统提示 / CLAUDE.md / AGENTS.md 最常见的两种翻车,官方称之为高度(altitude)问题:

❌ 太低:硬编码脆弱逻辑如果用户说"退款"就回复模板A,如果说"退钱"回复模板B,如果说"取消订阅"… ——穷举 if-else 既写不全又难维护,模型沦为查表机。
❌ 太高:空洞口号请提供优质、专业、令人满意的客服体验 ——没有任何可执行信号,等于没写。
✅ 正确高度:给启发式,不给穷举"退款类请求:先确认订单号与购买渠道;7 天内无理由直接受理;超期的说明政策并给补偿选项;任何时候不承诺你无法验证的时限。"——具体到能指导行为,又留了判断空间。

原则③ 位置策略与检索节奏

  • 首尾效应:关键指令放上下文开头或结尾,大块材料放中间;
  • Just-in-time 检索:智能体场景下,别把可能用到的文件全预载——给模型检索工具,让它按需去取(轻量标识符 + 运行时加载),这是 Claude Code 等编码代理的默认哲学;
  • 2026 平台能力:上下文编辑(自动裁剪过期工具结果)、记忆工具(跨会话持久存储)等已进入官方 API 能力面,重活正在从提示词层下沉到平台层——以官方文档为准。

来源:Anthropic:Effective context engineering for AI agents(注意力预算/高度/最小高信号集的原始出处)

进阶 04

智能体上下文:让长任务不"失忆"

当任务从"一问一答"变成"自主跑几十分钟",上下文管理从技巧升级为架构问题。三个层面:

① 常驻指令怎么写:CLAUDE.md / AGENTS.md 设计法

原则做法
只留"猜不出来的"构建命令、隐藏约定、踩过的坑;读代码就能知道的不写
短到极致每个会话都加载=每个 token 都全程收租;超过一屏就该拆
规则分流必须 100% 执行的 → Hooks/权限(强制);场景性知识 → Skills(按需加载);只有"每次都要的少量事实"才配常驻
高度正确用 03 章的高度原则写:启发式+红线,不写 if-else 也不喊口号

两份可直接抄的成品模板:转型手册的 CLAUDE.md 模板 · Codex 指南的 AGENTS.md 精瘦模板。

② 工具与示例也是上下文

  • 工具描述即提示词:模型靠工具的名字和描述决定何时调用——描述含糊的工具会被误用或冷落;工具集要最小可行,功能重叠的工具让模型选择困难(对应 CC 纪律:MCP 只装用得上的);
  • 示例宁缺毋滥:给智能体的示例同样遵循"多样典范"原则,堆边缘 case 只会教坏它。

③ 长任务三板斧(超出窗口怎么办)

板斧机制对应工具动作
压缩 Compaction把陈旧对话蒸馏成摘要,腾出窗口继续跑CC 的 /compact;平台的上下文编辑
结构化笔记把关键状态写到窗口外的持久载体,需要时再读回让代理维护 NOTES.md / TODO 清单;记忆工具
子代理架构探索交给独立上下文的分身,主窗口只收结论摘要CC 子代理;Codex 云端任务
一条判断线官方建议的次序:先做最简单可行的——窗口够用就好好修剪;真不够,压缩优先;要跨会话记忆,上笔记;探索量巨大,才上子代理。别为了架构而架构。

来源:同 03 章(压缩/笔记/子代理的原始论述) · CC 指南 · 上下文管理进阶(工具层实操)

实战 05

逐字模板库(9 式,复制改词即用)

覆盖高频场景。每式都内嵌了前四章的技法——照抄之后,建议对照标注回看"为什么这么写"。

① 万能任务骨架(G-C-C-D)
Goal: 一句话说清要产出什么 Context: 背景材料/相关文件(用标签围起来) Constraints: 不许做什么、必须遵守什么 Done-when: 什么样算完成(可验证的标准)

Codex 官方 Best Practices 推荐结构,通用于一切工具。

② 深度改稿(反向提纲式)
<draft>…我的初稿…</draft> 先为这篇初稿列反向提纲(每段实际在说什么),指出逻辑断层、重复 和"该说没说"的缺口;不要改写,先只做诊断。我确认后再动笔。
③ 调研快查(带溯源)
调研问题:__。范围:只看 2025 年后的信息;排除 __。 产出:结论(带置信度)+ 关键证据清单(每条附来源链接)+ 主要分歧点。规则:找不到可靠来源的直接说没找到,不要推测。
④ 学习私教(苏格拉底式)
我想真正理解 __。不要直接讲解:先出 3 道题摸底, 根据我的回答定位薄弱点,再用我熟悉的 __ 领域做类比讲解, 最后出一道应用题验收。全程发现我理解错就立刻打断。
⑤ 代码功能开发
实现 __。相关文件:@a.ts @b.ts。约束:遵守项目 CLAUDE.md, 不引入新依赖。Done-when: 通过 npm test 且新增逻辑有测试。 先给实现计划,我确认后再动手。
⑥ 审查式把关(给 AI 的产出装刹车)
审查你刚才的产出:按 ①事实与引用 ②逻辑漏洞 ③遗漏的重要 情形 三类各列出问题(没有就明说),给每条一个修复建议。 最后回答:如果这是你自己的名誉背书,你会直接交付吗?
⑦ 会议/长文提炼
<transcript>…原文…</transcript> 输出三张清单:1) 已定决策 2) 行动项(负责人+期限,缺则标未定) 3) 遗留争议。保留所有数字与日期原文,不要改写口径。
⑧ 数据分析(代码优先)
<data>…或文件路径…</data> 问题:__。要求:用代码计算而不是口算;先输出你打算怎么算 (口径、过滤条件),我确认后执行;结论区分"数据支持的"和 "你的推测",推测必须标注。
⑨ 两次纠错重开(止损式)
这个问题我们已经两轮没修好。忘掉之前的尝试路径,现在: 1) 重新陈述问题与已排除的假设 2) 列 3 个新方向按可能性排序 3) 给每个方向一个 5 分钟可完成的最小验证实验。

模板与本库其他专题联动:编程场景细化见转型手册提示词库,设计场景见Pencil 实践范例集,场景选择见心法页收益矩阵。

实战 06

评估与迭代:别凭感觉改提示词

大多数人改提示词的方式是"感觉不对就加一句"——三个月后攒出一个 2000 字的缝合怪,没人知道哪句还在起作用。工程化的做法只需三步:

  1. 攒失败集:每次 AI 产出不合格,把「输入 + 差产出 + 你期望的产出」存进一个文档——这就是你的测试集。10 条起步就够用。
  2. 改一处,跑全集:每次只改提示词的一个变量,把失败集全部重跑,看修好了几条、弄坏了几条——修好 3 条弄坏 2 条不算进步。
  3. 定期减法:每月挑一次,逐段删掉提示词里的句子重跑失败集——删了不变差的,说明是废话,永久删除。提示词和 CLAUDE.md 一样,短是战斗力。
三条止损纪律(与心法页共识一致)① 同一问题纠正两轮无效 → 别第三轮,用模板⑨重开;② 高风险产出必过模板⑥的审查式把关;③ 对"提示词玄学"保持警惕——任何声称"加这句话质量暴涨"的偏方,先过你自己的失败集再信。

方法论依据:官方 evals 思想的个人化简化;止损纪律出处见心法页七条。

实战 07

反模式图鉴:流传最广的提示词偏方

反模式为什么错 → 正解
提示词膨胀出一次错加一条规则,最终指令互相冲突、彼此稀释。正解:用 06 章的减法机制,信噪比优先
头衔堆砌"你是世界顶级天才专家"不含任何信息量。正解:角色具体到岗位+立场+口径(技法⑤)
威胁/贿赂迷信"答错就惩罚你""给你 100 美元小费"——社区流传的都市传说,对现代模型无稳定收益,白占 token。正解:把这份心思花在写清 Done-when
边缘 case 塞满示例教会了模型钻牛角尖,常规输入反而答歪。正解:2-5 个多样典范示例(技法②)
一个会话当垃圾场翻译完接着调代码再顺便写周报——上下文互相污染,越用越笨。正解:一事一会话;跑偏即重开
把"请一步步思考"当万能药对简单任务只增加延迟;对复杂任务不如给出具体分析框架(技法③的 ✅ 写法)
用对话修数据让 AI"重新数一遍"数不对的数字。正解:让它写代码算(模板⑧)——口算不可靠是模型特性,不是提示词能修的
禁令式否定滥用整页"不要 X 不要 Y"却不说要什么。正解:每条禁令配一条"应该怎样"的正向指令
鉴别偏方的通用方法问三个问题:① 官方文档写过吗?② 有可复现的对比证据吗?③ 它增加了信息量还是只增加了字数?三问全挂的,就是玄学。
